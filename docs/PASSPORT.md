## B. Программа «всё в БД + связки через БД» (приоритет founder #1)

Цель: убрать контент из TS-бандла в D1 и связать сущности **внешними ключами**, а не строками. Делается **вертикальными срезами** (таблица → миграция данных из TS → API-роут → переключение клиента с бандла на fetch + состояния загрузки → тест), чтобы не ломать рабочее.

Порядок (от низкого риска к высокому):

1. **✅ `book_catalog` + связь автор→`entities` — СДЕЛАНО.** 131 книга в D1 (`book_catalog`), `author_entity_id` = FK на `entities` (113 связей, 0 битых). Роут `/api/books/catalog` отдаёт каталог; клиент (`BooksHub`/`SearchScreen`/`HomeScreen`) читает через гидрацию `bookCatalog.ts` с фолбэком на бандл. Миграция в репо: `apps/api/migrations/0011_book_catalog.sql`. Попутно исправлен дубль id `navadvipa-dhama-mahatmya` в `books.ts`.
2. **✅ `kirtan_artists` + `kirtan_albums` + связи — СДЕЛАНО.** 12 исполнителей + 7 альбомов в D1. Альбом→исполнитель (FK `artist_slug`), исполнитель→личность (FK `entity_id`: Прабхупада, Индрадьюмна Свами, Сачинандана Свами; 0 битых FK). Роут `GET /api/kirtans`; клиент через `kirtansHydrate.ts` (`useKirtans`) с сид-фолбэком; хелперы и воркер читают сид (идентичен БД). Миграция: `apps/api/migrations/0012_kirtans.sql`. (Аудио всё ещё у 3/7 альбомов — отдельная задача заливки в IA.)
3. **✅ `dhamas`/`dhama_clusters`/`tirthas`/`tirtha_persons` + связи — СДЕЛАНО.** 3 дхамы, 14 кластеров, 54 тиртхи в D1. Связи: тиртха→дхама (FK), тиртха→кластер, тиртха→личность (`tirtha_persons` — 130 связей, 126 разрешены в `entities`; Шачи и Джаганнатха-группа оставлены NULL, без догадок), дхама→первичное Божество (FK `deity_entity_id`). 0 битых FK. Роут `GET /api/dhamas` (вложенная структура); клиент через `dhama/dhamasHydrate.ts` (`useDhamas`) с сид-фолбэком. Миграция: `apps/api/migrations/0013_dhamas.sql`. (Связь дхама→`places` городов — отдельный шаг.)
4. **`recipes` (прасадам) + связи рецепт→фестиваль (`gcal`/`festivals`) и рецепт→Божество.** Перенести `cookbook.ts`.
5. **`shop_catalog`** — перенести `catalog.ts`; товар-книга → FK на `book_catalog`.
6. **Связать `places` (956) с сущностями** — presiding Божество/основные личности места (матчинг по именам + ручная выверка).
7. **Чистка:** убрать пустую `entity_citations` либо наполнить из longform.
8. **Воспроизводимость:** все эти DDL/сиды положить **миграциями в репо** (`apps/api/migrations/`), закрыть дрейф схемы.

Принципы (из практики проекта): UTF-8 пишем через Python; большие вставки — батчами ~20 строк; правки клиента — с состояниями загрузки и фолбэком; App.tsx — union-merge.

> Эти срезы НЕ делаются «за один проход» безопасно — каждый меняет и бэкенд, и клиент. Выполняются по одному с проверкой, чтобы не повторить регрессий ленты.
