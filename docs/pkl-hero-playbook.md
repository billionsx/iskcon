# ЗАКОН ПРОЕКТА · Playbook сборки карточки личности (ПКЛ)

> **Обязательная схема для КАЖДОЙ личности.** Шастра-источник меняется (Чайтанья-чаритамрита, Шримад-Бхагаватам, Бхакти-ратнакара, Према-виласа, Гаура-ганоддеша-дипика, Прабхупада-лиламрита и т.д.) — но **суть, глубина проработки, авторитетность (BBT-grade, дословность, источник), атрибуция говорящего, тематические блоки, табы/суб-табы и цикл аудита НЕ меняются.**
>
> Меняется только «паспорт источника» (Фаза 0). Всё остальное — инвариант.
>
> **Эталонная реализация:** карточка `rupa-goswami` — 56 секций, 260 цитат, 4 таба / 16 суб-табов, полный охват ЧЧ (стихи + субстантивные комментарии), 0 битых ссылок, 0 дублей.

## Закон имён и терминов (обязателен во всех авторских полях)

Полные законы: `docs/STANDARD_canonical_names.md`, `docs/doctrine/gauranga-lila.md`, `docs/STANDARD_scripture_terms.md`. Применяются к КАЖДОЙ карточке.

**Как рендерится longform (`EntityPage.tsx`) — что это значит для данных:**
- Текст цитаты `q.t` и связка `p` идут через `renderTerms` → **термины авто-курсивятся** глоссарием (`src/ui/scripture.ts`, `TERMS`). Данные пишем **плоским текстом**, курсив руками не размечаем. Если термин не курсивится — добавить его форму в `TERMS`.
- Подпись говорящего `by`, ярлыки `see.t`, заголовки `h`, кикеры — рендерятся **плоско** (имя показывается буквально). Здесь **обязателен канон имён**.

**Канон имён (в `by` / `see` / `h` / `kicker` / `p`):**
- Господь → **«Гауранга Махапрабху»** (Навадвипа-лила, общее) или **«Шри Кришна Чайтанья Махапрабху»** (после санньясы — Пури, Нилачала). Весь контент про Госвами — пост-санньяса (они встретили Господа уже санньяси), поэтому подпись Господа = **«Шри Кришна Чайтанья Махапрабху»**. Голого «Чайтанья»/«Шри Чайтанья» — нет.
- **«Шримати Радхарани»** (не голое «Радхарани»), **«Шрила Прабхупада»** (не голое «Прабхупада»), **«Кришна»** как есть.
- **«Гауранга Лила»**, **«Кришна Лила»** — оба слова с заглавной, без дефиса; «Гаура-лила» не используется.

**Освобождены от канона имён** (не редактируем никогда): текст цитат `q.t` и выдержки Прабхупады — прямые цитаты из БД (даже если внутри «Шри Чайтанья Махапрабху»); названия книг («…-чаритамрита», «Чайтанья-чандродая»); slug/id.

**ИМЯ САМОГО ГЕРОЯ — ВСЕГДА ПОЛНОЕ В АВТОРСКИХ ПОЛЯХ (безусловное правило, 7.07.2026).** В `p` / `h` / `label` / `see.t` / `by` / `kicker` имя личности пишется ТОЛЬКО в полной форме с титулом — «Рупа Госвами», «Санатана Госвами», «Джива Госвами», «Гададхара Пандит», «Шриваса Тхакур», «Рамананда Рай», «Сварупа Дамодара Госвами» и т.д., во ВСЕХ падежах («Рупы Госвами», «Рупе Госвами»…). **Голая краткая форма («Рупа», «Санатана», «Джива», «Гададхара», «Рамананда») в авторском тексте ЗАПРЕЩЕНА** — даже в потоковой прозе, даже если так пишет БиБиТи. **Единственное исключение** — дословная цитата из ЧЧ в поле `q.t` (например, Махапрабху обращается к герою по имени): цитаты не правятся никогда.
- **Ловушки-коллизии (не испортить данные!):** «Гопала» — это и имя Кришны → титуловать ТОЛЬКО «Гопала Бхатта»→«Гопала Бхатта Госвами», голое «Гопала» (Кришна, «Гопала-чампу») не трогать. «Рагхунатха» неоднозначно (дас / Бхатта / Пури) → приводить к верной полной форме по контексту карточки. Защищать составные («Рупа-манджари», «Гопала-чампу», «Шриваса-ангана», «Санатана-шикша») и панча-таттва-мантру. «Сварупа» (заглавная С) ≠ «рупа».

**Термины (курсив по BBT):** `бхакти`, `дхарма`, `према`, `санкиртана`, `джапа`, `парампара`, `садхана`, `бхава`, `самбандха`, `мадхукари`, расы и т.д.; составные (`карма-йога`, `маха-мантра`) — целиком. Прямым — имена собственные и лоанворды (`йога`, `карма`, `гуру`, `мантра`, `аватара`).

## 0. НЕИЗМЕННЫЕ ПРИНЦИПЫ (инварианты поверх любого источника)

1. **Дословность.** Текст стиха и комментария — точно из БД, без пересказа и без правок. Единственная допустимая правка — нормализация прямой кавычки `"` в «ёлочки» ради валидного JSON; смысл не менять.
2. **Нулевая фабрикация.** Ничего не выдумывать. При нехватке данных — честный минимум, а не изобретённые детали.
3. **Авторитетность.** Каждая цитата снабжена тремя вещами: источник (`ref`), ссылка на **конкретный стих** (`to`), и **автор слов** (`by` + `byId` → личность).
4. **Полнота.** В карточке — ВСЕ сюжетные и тематические стихи о герое и ВСЕ субстантивные комментарии из всего источника. Вне карточки допустимо только формульное: подписи-строфы глав, призывы, колофон/оглавление.
5. **Структурность.** Стихи не «простынёй»: тематические блоки (заголовок + связка + относящиеся стихи), разложенные по табам → суб-табам.
6. **Проверяемость.** Каждая фаза заканчивается контролем; итог проходит gate-критерии; при провале — возврат в цикл.

## Форматы данных (эталон — карточка `krishna`)

- **Блок стиха:** `{ "t":"<дословный перевод>", "ref":"<источник>", "to":"<ссылка на стих>", "by":"<имя говорящего>", "byId":"<id личности>" }`
- **Секция:** `{ "h":"<заголовок>", "p":["<связующая фраза>"], "quotes":[<блоки стихов>], "see":[{"id","t"}] }`
- **Карточка (longform):** `{ "tabs":[ { "id","label","kicker","subtabs":[ { "id","label","sections":[<секции>] } ] } ] }`
- **Хранилище:** `entity_profiles.longform` (JSON), запись через MCP `d1_database_query`.
- **Прод D1:** `6226aded-dd03-4e74-977f-9cd0b509e73d`. Таблицы: `verses` (v.ref, v.division_id, v.work_id, v.ordinal), `verse_texts` (vt.verse_id, vt.translation, vt.purport), `entities` (id), `entity_profiles` (entity_id, longform).

## ФАЗА 0 — ПАСПОРТ ИСТОЧНИКА (единожды на источник)

Определить и зафиксировать параметры, от которых зависят все дальнейшие запросы. **Без этого генерировать нельзя.**

- **`:WORK`** — id источника (`cc` / `sb` / `brk` / …).
- **`:REF`** — формат строки ссылки (напр. «ЧЧ Мадхья 1.166», «ШБ 1.2.3»).
- **`:DIVISION`** — как `division_id` кодирует песнь/лилу/главу (напр. `cc.madhya.1`, `sb.1.2`).
- **`:ROUTE`** — маршрут ридера И **проверить вручную, работает ли ссылка на КОНКРЕТНЫЙ стих.** Для ЧЧ `/book/cc/{canto}/{ch}/{verse}` работает. Если у источника ридер только на главу — ссылаться на главу.
- **`:NARRATOR`** — автор/рассказчик источника = **дефолтный `by` для повествования**:
  - ЧЧ → Кришнадас Кавираджа (`krishnadasa-kaviraja`)
  - ШБ → Шукадева Госвами (`shukadeva`); где уместно — Сута Госвами или Вьяса
  - Бхакти-ратнакара → Нарахари Чакраварти
  - (для каждого источника уточнить точный id в `entities`.)
- **`:EXCLUDE`** — формульные паттерны источника, которые НЕ берём:
  - ЧЧ: строфа-подпись «Молясь у лотосных стоп Шри Рупы и Шри Рагхунатхи…» (повторяется в ~55 главах), призывы к шести Госвами, колофон/оглавление.
  - Другой источник — свои формулы (мангалачарана, колофоны глав и т.п.). Выявить в Фазе 1 и внести сюда.
- **`:ALIASES`** — до-санньясные/мирские имена героя и их склонения (Рупа = Дабир Кхас, Сакар Маллик; и т.д.). Учитывать при поиске стихов и комментариев.
- **`:SPEAKERS`** — набор говорящих, релевантных источнику, с подтверждёнными id в `entities` (для CASE-атрибуции в Фазе 4).

```sql
SELECT COUNT(*) verses, SUM(vt.purport IS NOT NULL) with_purport, MIN(v.ref) ref_ex, MIN(v.division_id) div_ex
FROM verses v JOIN verse_texts vt ON vt.verse_id=v.id WHERE v.work_id=':WORK';
SELECT id FROM entities WHERE id IN (':HERO','prabhupada',':NARRATOR', <кандидаты-говорящие>);
```

## КОРПУС ИСТОЧНИКОВ — приложение · docs/sources · бхаджаны (расширение Фазы 0)

Личность обогащается из ТРЁХ классов источников. **Стандарт подписи ЕДИН для всех** (см. «Форматы данных»): `{t, ref, by, byId}`. Поле `to` (кликабельная ссылка на стих) — ТОЛЬКО когда источник есть в приложении; иначе `to` ОПУСКАЕТСЯ: рендер (`expandRef`/`SourceLink` в `EntityPage.tsx`) показывает ссылку академически, простым текстом, а `by`+`byId` остаётся кликабельным автором. Внесли книгу в приложение → `to` до-заполняется (ref-строки стабильны).

**A. Книги приложения (D1 `verses`).** ЧЧ, ШБ, БГ, НП, Брахма-самхита… — дословные цитаты, SQL-генерация (Фаза 4), `to` кликабельно. Как раньше.

**B. Библиотека `docs/sources` (GitHub, вне D1, плоский текст).** Работать `grep` по `*.txt` (см. `docs/sources/README.md`).
- **ПЕРВОИСТОЧНИК** — можно дословные цитаты. RU-текст (Чайтанья-мангала, Чайтанья-бхагавата, Матхура-мандала = 5-я Волна БР, Навадвипа-дхама-махатмья): дословный блок по-русски, `to` опустить. EN-текст (Бхакти-ратнакара, Нароттама-виласа, Чайтанья-чарита-маха-кавья, Кришна-бхаджанамрита, Narahari-Songs): факты — в ПРОЗУ карточки (своими словами, сверяя); дословный блок-цитата — ТОЛЬКО после внесения книги в приложение с нашим переводом (свой перевод не выдавать за дословную шастра-цитату).
- **ВТОРИЧНЫЙ** (Роузен, Капур, «Lives of the Saints», Прабхупада-шикшамрита) — ТОЛЬКО факты/ориентировка/поиск эпизодов/сверка. НИКОГДА не дословная цитата; ссылаться на первоисточник, который они пересказывают.

**C. Бхаджаны.** Авторитетные гаудия-вайшнавские бхаджаны (в приложении и ещё не внесённые: Нароттама даса, Бхактивинода, Narahari-Songs …). Дословный блок; `by` = автор, `byId` = его сущность; `ref` = «„Название“, куплет N». В приложении → `to`; иначе академически.

**Паспорт docs/sources (дополнение Фазы 0):** ref — «Бхакти-ратнакара, Волна N» · «Чайтанья-бхагавата, {Ади/Мадхья/Антья} N.M» · «Чайтанья-мангала, {кханда} N» · «Нароттама-виласа, глава N». NARRATOR: Бхакти-ратнакара/Нароттама-виласа/Кришна-бхаджанамрита → Нарахари Чакраварти; Чайтанья-бхагавата → Вриндавана даса Тхакур; Чайтанья-мангала → Лочана даса Тхакур; Чайтанья-чарита-маха-кавья → Кави Карнапура (id сверять в `entities`, нет — не выдумывать). Поиск: кириллица по основе; санскрит в EN по латинскому стему без диакритики; учитывать мирские имена.

**Внесение книги/бхаджана в приложение — АЛГОРИТМ (решение 5.07.2026).** Триггер = нужда в ДОСЛОВНОЙ цитате. Нет дословной цитаты из книги → книгу НЕ вносим; факты идут в ПРОЗУ (перевод EN→RU в прозе разрешён, своими словами). Нашли дословную цитату для карточки → СРАЗУ вносим ЭТУ книгу в приложение ПОЛНОЦЕННО по стандартам (перевод на русский + разбивка книга/глава(волна)/стих, маршрут как ЧЧ/ШБ) → цитата ставится сразу КЛИКАБЕЛЬНОЙ (`to`). Бхаджан так же: нашли авторитетный бхаджан с отличной цитатой → вставляем цитату И запускаем добавление бхаджана по стандартам. Только ПЕРВОИСТОЧНИК/бхаджан; вторичные (Роузен/Капур/Lives) в приложение НЕ вносить — из них только проза-факты. Дословный книжный/бхаджанный блок живёт в карточке ТОЛЬКО когда источник внесён; «некликабельная дословная книжная подпись» не используется.

**ПОЛНОЕ ИССЛЕДОВАНИЕ ЛИЧНОСТИ — ТРИ ОБЯЗАТЕЛЬНЫХ ЦИКЛА (решение 7.07.2026, УЖЕСТОЧЕНО).** По КАЖДОЙ личности проводятся ТРИ полных цикла поиска и обогащения — ДО исчерпания ВСЕЙ достоверной и важной информации о личности, НЕ по капле, НЕ выборочно, НЕ «можно ещё углубить»:

- **ЦИКЛ 1 — ПРИЛОЖЕНИЕ.** Книги (стихи) + пурпорты Прабхупады (по ВСЕМ формам имени И тематически по сути) + параллельные пруф-стихи + ВСЕ бхаджаны. Каждое пророчество/цитату проверять на кликабельность.
- **ЦИКЛ 2 — БИБЛИОТЕКА.** `docs/sources`: grep-карта по ВСЕМ формам (EN+RU), прочитать КАЖДЫЙ источник с попаданиями, гомонимная дисциплина на КАЖДОМ совпадении.
- **ЦИКЛ 3 — ВНЕШНИЕ ДОСТОВЕРНЫЕ ИСТОЧНИКИ + ПЕРЕКРЁСТЫ + АУДИТ.** То, чего нет в корпусе (лиламриты, официальные биографии, Vedabase/Vaniquotes/BBT) — факты/даты/цитаты; двусторонние see-связи; финальный аудит.

Собрать ВСЁ по трём циклам → писать пачками → ОДИН аудит → ОДИН отчёт. **Категорически запрещено:** отчитываться после каждого источника; писать «можно добавить/углубить» вместо того, чтобы добавить; оставлять на «следующий круг» доступное сейчас. **Если пользователь вынужден гнать на новый круг вручную — это ПРОВАЛ скрипта.**

**ЦИКЛ 1 · Книги приложения (все) — стихи, пурпорты, бхаджаны.** Обязательные проходы:
   - **(а) Стихи.** По ВСЕМ читаемым книгам (`book_catalog`), где встречается герой (ЧЧ, ШБ, БГ …) — дословные цитаты через SQL-генерацию (Фаза 4), `to` кликабельно. ОМОНИМЫ: проверять контекст (в ЧЧ «Шриниваса» = Шриваса Тхакур, НЕ Шринивас Ачарья 2-й генерации; личности пост-лилы в ЧЧ как субъект отсутствуют — их «совпадения» в ЧЧ НЕ привязывать к ним).
   - **(б) Пурпорты Прабхупады — ПО ВСЕМ 38 655 стихам, не только по «своей» книге.** Прабхупада обильно цитирует ачарьев 2-й генерации в комментариях к ЧУЖИМ стихам (Нароттама — 145 пурпортов). ОБЯЗАТЕЛЬНЫЙ запрос по герою:
     ```sql
     SELECT COUNT(*) FROM verse_texts WHERE instr(coalesce(purport,''),'<Имя>')>0;
     SELECT v.ref, v.id, substr(vt.purport, max(1,instr(vt.purport,'<Имя>')-40),160) ctx
       FROM verses v JOIN verse_texts vt ON vt.verse_id=v.id
      WHERE instr(vt.purport,'<Имя точно>')>0 ORDER BY v.division_id, v.ordinal;
     ```
     Субстантивные пурпорты (оценка/учение/цитата песни героя) → выдержкой: `by='Шрила Прабхупада'`, `byId='prabhupada'`, `ref='<источник>, комментарий'`, `to='/book/.../стих'` (кликабельно в ридер). Отдельная секция «Шрила Прабхупада о <герое>».
   - **(в) ПО ВСЕМ ФОРМАМ ИМЕНИ — НЕ один запрос.** Прабхупада пишет имя в разных формах: номинатив + косвенные падежи + мирское имя + дефисные/пробельные варианты («Джаганнатха дас» / «Джаганнатха-даса») + титулы. Узкий одноформенный поиск ПРОПУСКАЕТ пурпорты (проверено многократно). Гонять несколько запросов разными формами; широкий поиск по основе + фильтр (напр. `instr(purport,'Джаганнатх')>0 AND instr(purport,'Бабаджи')>0`), затем каждое попадание сверять на гомонимию (Джаганнатха-Пури = МЕСТО, не Бабаджи).
   - **(г) ТЕМАТИЧЕСКАЯ ДОБЫЧА ПУРПОРТОВ (НЕ по имени, а по СУТИ) — ГЛАВНЫЙ шаг, добавлен 7.07.2026.** Искать в 38 655 пурпортах САМИ утверждения/пророчества/миссию/учение, связанные с героем, по КЛЮЧЕВЫМ ФРАЗАМ — они часто стоят дословно и годятся в КЛИКАБЕЛЬНУЮ цитату, даже если имени героя рядом нет. Примеры: пророчество «в каждом городе и деревне» → в десятках пурпортов (ЧЧ Ади 17.203, Мадхья 25.264 …); ШБ 4.22.42 прямо называет Бхактивиноду и Бхактисиддханту желавшими его осуществить. Для каждой темы/пророчества карточки гнать `SELECT verse_id, substr(purport,…) WHERE instr(purport,'<ключевая фраза>')>0`.
   - **(д) ПАРАЛЛЕЛЬНЫЕ ПРУФ-СТИХИ.** Один стих часто есть В ДВУХ книгах — проверять обе (krishna-varnam = ШБ 11.5.32 И ЧЧ Ади 3.52; пруф-стих аватары/пророчество цитируется и в ШБ, и в ЧЧ). Выбирать наиболее авторитетную/полную версию, вторую держать в уме для «см. также».
   - **(е) КЛИКАБЕЛЬНОСТЬ — приоритет над прозой.** Прежде чем оставить пророчество/цитату/утверждение ПРОЗОЙ, проверить: не стоит ли оно ДОСЛОВНО в читаемой книге приложения (`cc`/`bg`/`sb`). Стоит → цитировать КЛИКАБЕЛЬНО (`to`), проза оставляется лишь когда дословного вхождения в приложении НЕТ. Текст цитаты (`q.t`) свободен от крит.9 (дословное писание), поэтому даже «Сарасвати Прабхупада»/«Шри Чайтанья» внутри цитаты допустимы.
   - **Письма/лекции Прабхупады:** личных писем в корпусе приложения/библиотеки НЕТ → дословно цитировать НЕЛЬЗЯ (честно сообщить пользователю; предложить добавить корпус). Прабхупада-шикшамрита (библиотека) → только проза-факты.

**ЦИКЛ 1 · Бхаджаны — ВСЕ до единого (контроль числом).**
   - Найти в `prayers`+`prayer_verses` ВСЕ бхаджаны героя: и его СОБСТВЕННЫЕ сочинения (если автор), и прославления героя (пранамы/аштаки/песни разлуки). Поиск по слагу автора, имени, транслиту, склонениям, мирским именам.
   - **Дедупликация слагов:** одна песня часто под 2–3 слагами (`shri-`/`shree-`, `-habe`/`-ha-be`) — считать УНИКАЛЬНЫЕ песни ПО ТЕКСТУ куплета, не по числу слагов.
   - **Процитировать КАЖДУЮ уникальную песню** (тематическим суб-табом), НЕ подмножество. Контроль на выходе: `число процитированных бхаджанов = число уникальных песен героя в приложении`.
   - `t` тянуть ПРЯМО из `prayer_verses` (см. «Дословность-гарантия»), `ref`=«Название» (куплет N), `to`=слаг, `by`+`byId`=автор.

**ЦИКЛ 2 · Библиотека (все источники).** Карта плотности: grep-счётчик имени (EN+RU формы) по КАЖДОМУ `docs/sources/*.txt`. По каждому источнику с материалом — прочитать пассажи, извлечь факты, СВЕРИТЬ с карточкой (`instr(longform,'…')`) на дубли, внести НЕдублирующее. Первоисточники → проза (перевод EN→RU) + дословно ТОЛЬКО после внесения книги; вторичные (Роузен/Капур/Lives/Прабхупада-шикшамрита) → только проза-факты, дословно НИКОГДА.
   - **ГОМОНИМНАЯ ДИСЦИПЛИНА на КАЖДОМ совпадении (обязательна).** Прежде чем внести материал — убедиться, что совпадение про ТУ личность, а не про тёзку/божество/место/однофамильца. Проверено: глава Капура «Гауракишор Широмани» (с.79) — ИНОЙ святой XVIII в., НЕ Гаура Кишора дас Бабаджи; «Джаганнатха-Пури» в пурпорте = место, не Джаганнатха дас Бабаджи; «Баладева» = и Господь Баларама (искать «Видьябхушан» для различения). Тематический источник (Прабхупада-шикшамрита = свод наставлений по темам, не биография) не путать с биографией: подтверждает факты, но нового жития не даёт.

**ЦИКЛ 3 · Внешние достоверные источники + перекрёстные связи (добавлен 7.07.2026).** Когда корпуса (приложение + `docs/sources`) не хватает для полноты — привлекать достоверные публичные источники: авторизованные лиламриты (Прабхупада-лиламрита Сатсварупы даса Госвами и т.п.), официальные биографии ISKCON, Vedabase/Vaniquotes, BBT-издания. Только для ФАКТОВ/ДАТ/ЦИТАТ/ПРОРОЧЕСТВ, которых нет в корпусе (напр., биография Прабхупады — её нет в `docs/sources`). Правила: спорные датировки — по стандарту BBT/ISKCON, с явной оговоркой в тексте («по одним источникам… по другим…»); факт и агиографию разделять; сомнительные писания («Чайтанья-упанишад», отдельные пуранические стихи) подавать формулой «принято в традиции Гаудия», не как бесспорный ведический канон; дословную книжную цитату из внешнего источника в карточку НЕ ставить (только проза-факты), пока книга не внесена в приложение по стандарту. **Перекрёстные связи:** каждая личность, упомянутая в карточке в контексте общего события/пророчества/парампары, получает `see`-ссылку; связи ДВУСТОРОННИЕ (если A ссылается на B по общему сюжету — добавить и B→A).

**3. Пачкой, отчёт ОДИН раз.** Сперва собрать ВЕСЬ материал по трём циклам; писать секции ПАЧКАМИ; аудит ОДИН раз в конце. Финал: URL карточки + «Обнови жёстко». НЕ отчитываться после каждого источника/капли.

**4. Дисциплина + технические ловушки (обязательны).**
   - **Дословность-гарантия (json_replace-фикс).** Ручной ввод цитаты рискует НЕВИДИМЫМ расхождением (прекомпозиция↔декомпозиция «ё», nbsp U+00A0) — текст визуально идентичен, но `instr(longform, verse_text)=0`. ПОСЛЕ добавления каждой пачки бхаджанов/пурпортов проверять КАЖДУЮ цитату: `instr(longform,(SELECT verse_text/translation/purport …))>0`. Провал → чинить подстановкой из БД: путь взять `SELECT fullkey FROM json_tree(...) WHERE fullkey LIKE '%.quotes[%].t' AND instr(value,'<фрагмент>')>0`, затем `UPDATE … SET longform=json_replace(longform,'$.путь.t',(SELECT verse_text FROM prayer_verses WHERE slug=… AND ord=1))`. json_replace с ПРЯМЫМ подзапросом РАБОТАЕТ (в отличие от вложенного в json_object). Лучшее — вообще не хэндтайпить: тянуть `t` из БД.
   - **D1 мультизапрос.** Большой многопутёвый `json_insert` UPDATE + verify-SELECT с подзапросами в ОДНОМ вызове ПАДАЕТ («no such column: longform at offset …»). Правило: крупный UPDATE — ОТДЕЛЬНЫМ вызовом, verify — СЛЕДУЮЩИМ вызовом. Подзапрос как ЗНАЧЕНИЕ внутри `json_set/json_insert` тоже падает; `json_replace(longform,path,(подзапрос))` — исключение. Кириллица в поиске — `instr`, НЕ `LIKE`.
   - **Крит.9 — ложные срабатывания** (скан регистро- и падеж-чувствителен через `instr`): (а) панча-таттва-мантра «шри-кришна-чайтанья» строчными НЕ даёт заглавного «Кришн» → абзац с «Чайтанья» тонет; фикс «Если Чайтанья…» → «Если Шри Кришна Чайтанья Махапрабху…». (б) «Шри Чайтаньи Махапрабху» без «Кришн» → «Шри Кришны Чайтаньи Махапрабху». (в) родительный «Шрилы Прабхупады» НЕ содержит именительного «Шрила Прабхупад» → заголовок в именит. форму («Шрила Прабхупада о <герое>»). (г) имена с «Чайтань» (Чайтаньянанда, Чайтанья дас) тонут, если в их абзаце нет заглавного «Кришн» — переписать абзац с полной формой имени Господа.
   - **Омонимы/атрибуция:** не смешивать тёзок (Гададхара Пандит ≠ Гададхара дас ≠ Гададхара Бхатта; Шриваса ≠ Шринивас Ачарья); не привязывать чужую лилу.
   - **Дубли заголовков:** `total_h=distinct_h`; дубль — слить в существующую секцию (перенести содержимое+`cite`), копию убрать.
   - **Канон имён** (титулы «Госвами/Свами/Махарадж/Прабху/Тхакур») + сверка КАЖДОГО термина с глоссарием `scripture.ts`, недостающие вносить.

## ФАЗА 1 — ПОЛНЫЙ СБОР СТИХОВ

```sql
SELECT v.ref, substr(vt.translation,1,80) tr
FROM verses v JOIN verse_texts vt ON vt.verse_id=v.id
WHERE v.work_id=':WORK'
 AND ( instr(vt.translation,'<Имя>')>0 OR instr(vt.translation,'<Алиас1>')>0 /* … :ALIASES … */ )
 AND instr(vt.translation,'<формула-подпись>')=0   /* :EXCLUDE */
ORDER BY v.division_id, v.ordinal;
```

Разложить результат:
- **сюжетные** (жизнь, деяния, речь героя/о герое) — берём ВСЕ;
- **тематические** (герой как автор, образец, объект прославления) — берём;
- **формульные** (подпись глав, призывы, колофон/оглавление) — НЕ берём (в `:EXCLUDE`).

Составить упорядоченный по эпизодам список `ref`.

> Предупреждение: перевод стиха может НЕ называть героя (например, реплики в диалоге), но по сути быть его словами. Такие стихи поиск по имени не берёт — при необходимости добавить их вручную отдельным суб-табом (напр. «Разбор рас»), пометив как расширение.

## ФАЗА 2 — АТРИБУЦИЯ «АВТОР СЛОВ → ЛИЧНОСТЬ»

Для каждого `ref` определить `by`/`byId` по правилу **чьи это слова**:
- **повествование** (рассказчик описывает события) → `:NARRATOR`;
- **прямая речь как основа стиха** → говорящий (Господь → `chaitanya`; герой → `:HERO`; другой участник → его id);
- **цитата чужого труда внутри стиха** → автор того труда (напр. `kavi-karnapura`);
- стих-повествование с крошечной вставной репликой → `:NARRATOR`.

Собрать по говорящим списки `ref` (для CASE). Всё, что не в списках, → `:NARRATOR` по умолчанию.

## ФАЗА 4 — ГЕНЕРАЦИЯ БЛОКОВ (SQL из БД, НЕ ручной набор)

Генерируй `quotes` прямо из БД — гарантирует **точный текст**, **вычисленную ссылку на стих** и **авто-экранирование прямых кавычек** (снимает поломку JSON). Первый блок карточки — REPLACE (`SET longform=json_array(...)`), остальные — append (`json_insert(longform,'$[#]',...)`).

```sql
UPDATE entity_profiles SET longform=json_insert(longform,'$[#]', json_object(
 'h','<Заголовок>',
 'p',json_array('<Связующая фраза>'),
 'quotes', json((SELECT json_group_array(json_object(
    't',  vt.translation,
    'ref', v.ref,
    -- :ROUTE — ссылка на конкретный стих. Ниже — разбор division_id вида 'cc.canto.chapter' + номер из ref.
    -- ДЛЯ ДРУГОГО ИСТОЧНИКА адаптировать под его division_id/ref/route.
    'to', '/book/'||':WORK'||'/'
          ||substr(v.division_id, length(':WORK')+2, instr(substr(v.division_id,length(':WORK')+2),'.')-1)||'/'
          ||substr(v.division_id, length(':WORK')+2+instr(substr(v.division_id,length(':WORK')+2),'.'))||'/'
          ||substr(v.ref, instr(v.ref,'.')+1),
    'by',  CASE WHEN v.ref IN (<refs речи Господа>) THEN 'Шри Чайтанья Махапрабху'
                WHEN v.ref IN (<refs речи героя>)   THEN '<Имя героя>'
                ELSE '<:NARRATOR имя>' END,
    'byId',CASE WHEN v.ref IN (<refs речи Господа>) THEN 'chaitanya'
                WHEN v.ref IN (<refs речи героя>)   THEN ':HERO'
                ELSE ':NARRATOR' END
  ) ORDER BY v.ordinal)
  FROM verses v JOIN verse_texts vt ON vt.verse_id=v.id WHERE v.ref IN (<refs блока>))),
 'see', json_array(json_object('id','<id>','t','<имя>'))
)) WHERE entity_id=':HERO';
```

После каждой пачки блоков — контроль:
```sql
SELECT json_valid(longform) jv, json_array_length(longform) blocks,
 (SELECT COUNT(*) FROM json_tree((SELECT longform FROM entity_profiles WHERE entity_id=':HERO')) WHERE key='byId') quotes,
 (SELECT COUNT(*) FROM json_tree((SELECT longform FROM entity_profiles WHERE entity_id=':HERO')) WHERE key='to' AND value NOT LIKE '/book/'||':WORK'||'/%') bad_links
FROM entity_profiles WHERE entity_id=':HERO';
```

## ФАЗА 5 — КОММЕНТАРИИ (пурпорты)

```sql
-- ВНИМАНИЕ: имена-омонимы. Подстрока имени героя может входить в другое имя
-- (пример ЧЧ: «Сва-РУП-а Дамодара» ловится поиском «Руп»). Ищи ТОЧНО по «<Имя> Госвами»/склонениям.
```

- **(А) цитаты книг героя** («это стих из „…“») → НЕ плодить обрывками; собрать **ОДИН сводный блок** «Труды как источник учения» — по каждой книге + характерная ссылка.

## ФАЗА 6 — ТАБЫ И СУБ-ТАБЫ

Снять индексы всех блоков, разметить по табам → суб-табам, перестроить `longform` одним запросом (читает старый flat-массив, пишет `{tabs}`; секции встраиваются через `json(json_extract(...))`).

```sql
-- карта индексов блоков:
SELECT j.key idx, json_extract(j.value,'$.h') h
FROM json_each((SELECT longform FROM entity_profiles WHERE entity_id=':HERO')) j
ORDER BY CAST(j.key AS INTEGER);

-- перестройка:
UPDATE entity_profiles SET longform = json_object('tabs', json_array(
  json_object('id','<t1>','label','<Таб 1>','kicker','<Kicker>','subtabs', json_array(
    json_object('id','<s1>','label','<Суб 1>','sections', json_array(
       json(json_extract(longform,'$[0]')), json(json_extract(longform,'$[1]')) /* … индексы блоков … */ ))
    /* … суб-табы … */ ))
  /* … табы … */
)) WHERE entity_id=':HERO';
```

Правило разметки: **каждый индекс блока используется РОВНО один раз** (проверяется в Фазе 7). Число табов НЕ фиксировано — определяется объёмом материала ради лучшей навигации (табы возникают и убираются); суб-табы группируют по 2–5 секций как ориентир, не жёсткий предел. Порядок: житие → служение/наследие → труды → комментарии.

**ОБЯЗАТЕЛЬНЫЙ ПЕРВЫЙ ТАБ «Вклад в Гауранга Лилу» (id `vklad-gauranga-lila`, добавлен 7.07.2026; ПЕРВЫМ — уточнено 7.07.2026).** В КАЖДОЙ карточке личности этот таб идёт **ПЕРВЫМ** (`$.tabs[0]`) — ибо по сути это вывод и смысл всей жизни героя, а не приложение в конце. Это отдельный таб-синтез: как жизнь, действия и труды героя ПРЯМО ИЛИ КОСВЕННО **готовили и служили миссии Шри Кришны Чайтаньи Махапрабху — Гауранга Лиле** — и как они стали **основой, помощью, вдохновением** во вкладах и жизнях других ачарьев и садху. Наполнение — строго из материала трёх циклов, без фабрикации:
- **Прямые спутники лилы** (были В Гауранга Лиле): их непосредственное участие + что они сохранили и передали для её продолжения (записи-качи, наставления, установленные Божества и святые места, обученные ученики). Пример: Сварупа Дамодара — хранитель сокровенной Пури-лилы, чьи записи легли в основу «Чайтанья-чаритамриты».
- **Позднейшие ачарьи** (после ухода Господа): чем их труд/институт/проповедь/печать **подготовили, возродили, распространили или сберегли** волну Гауранга Лилы (ср. табы «Пророчества»/«Парампара» у линии Бхактивинода → Бхактисиддханта → Прабхупада).
- **Связи-влияния:** на чей труд опирался герой и кого вдохновил — с ДВУСТОРОННИМИ `see`-ссылками на этих ачарьев. Каждое утверждение — на фактах из циклов; общий тезис «все ачарьи приходили ради Гауранга Лилы» раскрывается через КОНКРЕТНЫЙ вклад ИМЕННО этого героя, а не лозунгом.

## ФАЗА 7 — АУДИТ (ЦИКЛ ДО НУЛЯ)

```sql
-- 7.1 Целостность структуры + дубли + сохранность
SELECT json_valid(longform) jv, json_array_length(longform,'$.tabs') tabs,
 (SELECT COUNT(*) FROM json_tree((SELECT longform FROM entity_profiles WHERE entity_id=':HERO')) WHERE key='sections') subtabs,
 (SELECT COUNT(*) FROM json_tree((SELECT longform FROM entity_profiles WHERE entity_id=':HERO')) WHERE key='h') sections,
 (SELECT COUNT(*) FROM json_tree((SELECT longform FROM entity_profiles WHERE entity_id=':HERO')) WHERE key='byId') quotes,
 (SELECT COUNT(*) FROM json_tree((SELECT longform FROM entity_profiles WHERE entity_id=':HERO')) WHERE key='ref') total_refs,
 (SELECT COUNT(DISTINCT value) FROM json_tree((SELECT longform FROM entity_profiles WHERE entity_id=':HERO')) WHERE key='ref') distinct_refs,
 (SELECT COUNT(*) FROM json_tree((SELECT longform FROM entity_profiles WHERE entity_id=':HERO')) WHERE key='to' AND value NOT LIKE '/book/'||':WORK'||'/%') bad_links
FROM entity_profiles WHERE entity_id=':HERO';

-- если total_refs != distinct_refs → найти и убрать дубль:
SELECT value, COUNT(*) c FROM json_tree((SELECT longform FROM entity_profiles WHERE entity_id=':HERO'))
 WHERE key='ref' GROUP BY value HAVING COUNT(*)>1;

-- 7.2 Полнота стихов: что НЕ в карточке (должны быть только подпись/призывы/колофон)
SELECT COUNT(*) not_in,
 SUM(CASE WHEN instr(vt.translation,'<формула-подпись>')>0 THEN 1 ELSE 0 END) formulaic
FROM verses v JOIN verse_texts vt ON vt.verse_id=v.id
WHERE v.work_id=':WORK' AND instr(vt.translation,'<Имя>')>0
 AND v.ref NOT IN (SELECT value FROM json_tree((SELECT longform FROM entity_profiles WHERE entity_id=':HERO')) WHERE key='ref');
-- «прочее» (не формула) вывести списком и глазами убедиться: только призывы/оглавление.

-- 7.3 Полнота комментариев: остались ли субстантивные (не цитаты книг)?
--     точный поиск по имени + фильтр маркеров цитат ('стих из','вошел в','цитата из','пишет','из «', …);
--     если результат непуст и содержит биографию/даты/роль → вынести выдержкой (Фаза 5).
```

**Условия возврата в цикл:**
- `bad_links>0` или `total_refs≠distinct_refs` → Фаза 4/6 (починить генерацию/разметку, убрать дубль).
- в 7.2 «прочее» содержит сюжетный стих → Фаза 1–4 (добавить).
- в 7.3 нашлась субстантивная выдержка → Фаза 5 (добавить блок).
- число секций/цитат «до» и «после» реструктуризации не сходится → Фаза 6 (потеря/дубль при разметке).
- Иначе — **выход из цикла.**

## GATE — КРИТЕРИИ ИДЕАЛЬНОЙ КАРТОЧКИ (условие выхода)

1. `json_valid=1`; структура `tabs → subtabs → sections`.
2. Все сюжетные и тематические стихи героя из всего источника; вне карточки — только формульные.
3. Текст стихов и комментариев **дословно из БД**; `to` — на конкретный стих; `bad_links=0`.
4. `by`/`byId` — **говорящий слов** со ссылкой на личность.
5. Стихи в тематических блоках (заголовок + связка + стихи), не «простынёй».
6. Субстантивные комментарии — выдержками (`prabhupada`); цитаты книг — сводным блоком.
7. `total_refs = distinct_refs` (0 дублей); число секций/цитат сходится до и после реструктуризации.
8. Навигация организована табами и суб-табами; их число НЕ фиксировано, а определяется объёмом материала ради наилучшей навигации.
9. **Имена (авторские поля):** в `by`/`see`/`h`/`kicker`/`p` — 0 неканонических форм Господа/Радхарани/Прабхупады. Проверка: `json_tree` по `key IN ('by','kicker','h')` и `fullkey LIKE '%.p[%'` с `instr(value,'Чайтань')>0` — остаться должны лишь названия книг и канон «Шри Кришны Чайтаньи Махапрабху»; голого `by='Шри Чайтанья Махапрабху'` = 0. Термины — плоским текстом (курсив даёт рендер).
10. **Таб «Вклад в Гауранга Лилу» присутствует И идёт ПЕРВЫМ** (`json_extract(longform,'$.tabs[0].id')='vklad-gauranga-lila'`): синтез прямого/косвенного вклада героя в миссию Гауранга Лилы и во вклады/жизни других ачарьев и садху — на фактах трёх циклов, с двусторонними `see`-связями, без лозунгов и фабрикации.
11. **Имя героя нигде не голое в авторских полях** (`p`/`h`/`label`/`see.t`/`by`/`kicker`): 0 вхождений краткой формы без титула. Проверка: `json_tree` по этим полям (кроме `%.quotes[%].t`) — стем имени без полной формы и без составного-дефиса = 0. Единственное законное исключение — `q.t` (дословная цитата ЧЧ). Коллизии «Гопала»(Кришна)/«Рагхунатха»(дас/Бхатта) обрабатываются по контексту.

Финальный ответ пользователю всегда завершается: `Обнови жёстко: Cmd+Shift+R`.

## УРОКИ (зашиты в схему)

- **Генерация из БД > ручной набор** — точность текста + авто-экранирование прямых кавычек (снимает поломку JSON).
- **Поиск по имени ловит омонимы** (Сварупа при поиске «Руп») — искать точно по «Имя Госвами»/склонениям.
- **Строфа-подпись повторяется в десятках глав** — исключать, не тащить копии.
- **Учитывать до-санньясные/мирские имена** героя при поиске стихов и комментариев.
- **Ссылка ведёт на стих, а не на главу** (где ридер это поддерживает — проверить в Фазе 0).
- **После реструктуризации — сверять сохранность** (секции/цитаты/дубли до и после).
- **Аудит не однократный, а до нулевого остатка** (стихи + комментарии + целостность табов).
- **Атрибуция — «автор слов на личность»**: повествование → рассказчик источника; прямая речь → говорящий.
