## 0. Как читать этот документ

У каждого числа есть **происхождение**. Оно помечено:

| Метка | Что значит | Можно ли спорить |
|---|---|---|
| 📐 **ЗАМЕР** | Снято пиксельно с кадра в `refs/`. Адрес кадра указан | Нет. Это факт. Спорить можно только с методом замера |
| 🍎 **КАНОН** | Опубликованное значение Apple (HIG / системный токен) | Нет |
| ⚙️ **ВЫВОД** | Наш перевод замера/канона в CSS. Обоснование указано | Да — если предложен лучший перевод |
| 🕳 **ДОЛГ** | В референсах этого ещё нет. Число не выдумано, а **не снято** | — |

**Выдумывать число запрещено (ЗКН-БТ001).** Нет кадра — стоит 🕳, а не
правдоподобная цифра. Правдоподобная цифра хуже отсутствующей: её нельзя
проверить, и она расползается по коду как факт.

### Перепись, а не впечатление

Числа ниже сняты **счётом по всем 217 кадрам десяти продуктов**, а не взглядом
на несколько экранов. Впечатление нельзя перепроверить; перепись можно
перезапустить и получить то же:

```bash
python3 tools/assets/offload.py restore --class ios26-refs
python3 tools/assets/offload.py restore --class ios26-refs-2
python3 tools/ios26-census.py --src <папка с кадрами> --json docs/design/ios26/census.json
```

Сырой результат лежит рядом: **`census.json`** — по каждому продукту холст,
поверхности, толщина и цвет разделителя, шаг строки, врезка, плавающий бар,
акцент. Спор о числе решается перезапуском переписи, а не памятью.

### Как снимать замер, чтобы он не соврал

В таблице выше сказано: со ЗАМЕРОМ спорить можно только через метод. За один
день роспись Notes трижды упёрлась в один и тот же дефект метода, и все три
раза неверное число выглядело правдоподобно. Поэтому метод записан здесь, а не
держится в голове.

**1. Край берётся по материалу, а не по яркости.** Антиалиасинг и ореол стекла
дают около пикселя лишку, и этот пиксель решает споры. Брать надо точку, где
заливка пересекает 50 % между холстом и деталью, с долей пикселя. Что стоила
яркостная кромка: скругление «22.3 pt, не дуга» (на деле 24–26), низ панели
«21.3 / 28.3» (на деле 21.0 / 28.0), высота таб-бара «61.7 с разбросом» (на
деле 62.00 ровно).

**2. Единица замера кегля — литера, а не окно.** Прямоугольник, заданный
рукой, ловит соседний знак или срезает букву: «строка папки» вышла 48 px
вместо 43, потому что в окно попала иконка. Литера — связная область чернил,
её границы задаёт кадр. Где в строке есть знаки (меню), берётся мода высот по
всей панели: литеры кучкуются в две горки, знак в горку не попадает.

**3. У повторяющегося элемента меряется инвариант, а не следствие.** Низ
плавающей панели полгода числился «то 21.3, то 28.3» и объяснялся типом
экрана. Постоянна не кромка, а **ось**: 52.00 pt над низом экрана. Кластеров
не было — было два разных предмета, померенных за низ.

**4. Модель без соперника ничего не доказывает.** «Круг дал бы 19.3, а замер
даёт 24» — довод, у которого нет второй модели. Подгонять надо минимум двумя
и печатать ошибку обеих: дуга против суперэллипса, а не дуга против интуиции.

Метод держится инструментами, а не памятью:

| Инструмент | Что снимает | Как проверяет себя |
|---|---|---|
| `tools/ios26-corner.py` | скругление | подгоняет дугу И суперэллипс, печатает RMSE обеих |
| `tools/ios26-type.py` | cap-height | по литере или по моде панели; кегль отдаётся отдельно как ⚙️ |
| `tools/ios26-bar.py` | плавающая панель | считает ось, отсекает индикатор, клавиатуру и лист |
| `tools/ios26-battery.py` | всё сразу по всем кадрам | одна мерка на десять продуктов — числа становятся сравнимыми |
| `tools/ios26-states.py` | состояния нижнего слоя | разбирает слой на сегменты и считает элементы — механика видна там, где по одной ширине её нет |
| `tools/ios26-textcensus.py` | кегль, вес и цвет надписей | вес считается толщиной штриха, а не оценивается на глаз |

Адрес замера держит гейт, а не совесть: `tools/ios26-lint.py` проверяет, что
названный продукт есть в `refs/` **и что номер кадра не выходит за число кадров
продукта по реестру**. Ссылка на девятнадцатый кадр Notes, у которого их
восемнадцать, роняет прогон — раньше проходила молча: имя-то настоящее, потому
и незаметно.

Проверено это на живом нарушении, как и положено гейту: номер подменялся в
файле, прогон краснел, номер возвращался — зеленел. Заодно поймало и первую
редакцию этого абзаца: пример был написан настоящим адресом и был честно
отвергнут собственным правилом.

🕳 **Долг:** обратное — «у каждого 📐 есть адрес» — гейтом не берётся. Адрес
законно бывает трёх видов: ссылка на кадр, продукт, названный словами
(«тумблеры», «кольцо Move в Фитнесе»), и охват («88 кадров, 7 продуктов»).
Регулярка дала 24 находки, из них большинство — нормальная проза. Шумный гейт
отключают, и тогда не работает даже та часть, что работала.

### Порядок применения
```
1. Открыть этот документ  →  число есть?  →  взять его
2. Числа нет  →  открыть refs/, снять замер  →  вписать сюда с адресом кадра
3. Снять нечем (нет кадра)  →  пометить 🕳, запросить кадр у основателя
```
Шаг 3 не пропускается. «Сделаю по памяти Apple» — это шаг 0 из старой жизни.

## 1. Система координат

| Параметр | Значение | Источник |
|---|---|---|
| Кадр референса | 1179 × 2556 px | 📐 все кадры |
| Логический холст | **393 × 852 pt** | ⚙️ 1179/3 × 2556/3 |
| Масштаб | **@3x** → `pt = px / 3` | 📐 |
| Базовая сетка | **4 pt**; половинный шаг 2 pt допустим только внутри контрола | 🍎 |
| Минимальная зона нажатия | **44 × 44 pt** | 🍎 HIG |
| Пол типографики | **11 pt** | 🍎 + ЗКН-Д006 |

В CSS `1 pt = 1 px` (CSS-пиксель), потому что браузер уже работает в логических
единицах. **Все числа стандарта пишутся в CSS как есть.** Умножать на 3 нельзя —
это была бы вёрстка в физических пикселях.

Safe area:
```css
padding-bottom: max(20px, env(safe-area-inset-bottom));
padding-top:    env(safe-area-inset-top);
```
📐 Плавающий слой держится не отступом снизу, а **осью: 52.00 pt над низом
экрана** (88 кадров, 7 продуктов). Нижняя кромка — следствие оси и высоты
детали: таб-бар высотой 62 стоит в 21.0 pt от края, панель поиска высотой 48 —
в 28.0 pt. Ось в CSS не выражается, поэтому верстаем отступ той детали, которую
делаем, и не переносим её число на другую. Подробности — 5.2.

## 2. Цвет — тёмная тема

Тёмная тема iOS 26.5 стоит на **чистом чёрном холсте**, а не на тёмно-сером.
📐 `#000000` занимает **29–59 %** площади в каждом из шести приложений.
Это не стилистика — это OLED: чёрный пиксель выключен.

### 2.1 Холсты и поверхности

| Роль | Тёмная | Светлая | Источник |
|---|---|---|---|
| Холст экрана `systemBackground` | **`#000000`** | `#FFFFFF` | 📐 все кадры |
| Поверхность карточки `secondarySystemBackground` | **`#1C1C1E`** | `#F2F2F7` | 📐 `apple_id` с.4 (чтение `#1B1B1D`, JPEG −1) |
| Приподнятая поверхность `tertiarySystemBackground` | **`#2C2C2E`** | `#FFFFFF` | 📐 `apple_calendar` с.2, 8 (10.8 % площади) |
| Материал под баром / меню | **`#181818`** непрозрачный эквивалент | — | 📐 `apple_files` с.4 таб-бар |

📐 **Перепись 217 кадров.** `#000000` — модальный холст у **всех десяти**
продуктов без исключения (доля 29–59 %). Словарь поверхностей одинаков везде:

| Поверхность | Встречается у |
|---|---|
| `#000000` холст | 10 из 10 |
| `#1C1C1E` карточка | Calendar · ID · Wallet · App Store · Notes · Fitness |
| `#2C2C2E` приподнятая | Calendar · App Store · ID |
| `#181818` стекло над чёрным | 10 из 10 (бары, меню, листы) |
| `#121212` подложка медиа | TV · Music · Dev |
| `#3A3A3A`/`#3B3B3B` заливка контрола | Music · Notes · Fitness |

**Три уровня — потолок.** Четвёртый серый в тёмной теме неразличим: между
`#2C2C2E` и `#3A3A3C` глаз на OLED разницы не видит, а иерархия ломается.
Глубже — не цветом, а **тенью и материалом** (§6).

🍎 КАНОН. В тёмной теме вторичные метки — **не серый цвет, а белый с альфой**:

| Токен | Значение | Где |

### 2.4 Разделитель

| Параметр | Значение | Источник |
|---|---|---|
| Толщина | **1.0 pt** (3 физических px) | 📐 `apple_id` с.4 — ровно 3 px |
| Цвет поверх `#1C1C1E` | **`#37373A`** | 📐 три замера подряд: `#343436`, `#373739`, `#373739` |
| Канонический токен | `separator` = `rgba(84, 84, 88, 0.60)` | 🍎 — даёт `#37373A` над `#1C1C1E` ✓ |
| Непрозрачный | `opaqueSeparator` = `#38383A` | 🍎 |
| Врезка слева | **до начала текста** | 📐 68.7 pt от края экрана = 16 врезка + 16 поле + 29 плитка + 7.7 зазор |
| Врезка справа | **16 pt** от кромки карточки | 📐 линия кончается на 360.7 pt, карточка — на 377 pt |

📐 **Перепись:** толщина **1.0 pt** — модальная у восьми продуктов из десяти
(Calendar 108 · Dev 72 · Files 111 · ID 27 · Notes 74 · Wallet 22 попаданий).
Цвет над карточкой `#1C1C1E`: **`#36–38`** (`#373739` у ID, `#38383A` у Notes и
Wallet — это ровно `opaqueSeparator`). Над более светлой подложкой линия темнее
(`#292929`…`#2D2D2D`): iOS считает разделитель **альфой**, а не краской.

**0.5 px запрещён.** iOS рисует волосяную линию **в 1 pt**, а не в полпикселя;
полпикселя в браузере на 1x-экране исчезает совсем.

## 3. Типографика

### 3.1 Шрифт

```css
--font-ui:
  -apple-system, BlinkMacSystemFont,
  "SF Pro Text", "SF Pro Display", "SF Pro",
  system-ui, "Segoe UI", Roboto, "Inter", sans-serif;
```

🍎 На устройствах Apple `-apple-system` даёт настоящий SF и **сам** переключает
оптический размер: `SF Pro Text` до 20 pt, `SF Pro Display` от 20 pt.
Руками этот порог не эмулируют — это делает шрифт.

Вне Apple ближайшая метрика — **Inter**. ⚙️ Inter выше по x-height, поэтому при
подстановке кегль **не меняют**, а межстрочный держат по таблице ниже.

Цифры в счётчиках, времени, суммах:
```css
font-variant-numeric: tabular-nums;   /* иначе цифры прыгают при пересчёте */
```

```css
letter-spacing: calc(-0.43 / 17 * 1em); /* ≈ -0.0253em */
```

### 3.3 Вес несёт иерархию, а не важность

ЗКН-Д024 уже это сказал, референсы подтверждают: **заголовок строки списка —
17 обычным**, не полужирным. Полужирный в списке из шести строк делает
полужирными все шесть, и иерархии не остаётся.

Полужирный (600–700) — только:
крупный заголовок экрана · заголовок карточки · активный сегмент ·
кнопка-действие · заголовок алерта.

### 4.2 Скругления

📐 **Проверено на четырёх продуктах одним методом (19.07.2026).** Профиль угла
снимается субпиксельно — граница берётся там, где заливка пересекает 50 %
между холстом и карточкой, — и подгоняется МНК двумя моделями: дугой
окружности и суперэллипсом `|x/R|ⁿ + |y/R|ⁿ = 1`.

| Продукт | Дуга R | RMSE | Суперэллипс R | n | RMSE | Подгонок |
|---|---|---|---|---|---|---|
| `apple_id` | 24.17 pt | 1.03 px | 28.67 pt | **2.5** | 0.64 px | 3 |
| `apple_files` | 24.17 pt | 1.08 px | 28.67 pt | **2.5** | 0.64 px | 1 |
| `apple_notes` | 25.83 pt | 1.13 px | 31.33 pt | **2.6** | 0.69 px | 3 |
| `apple_wallet` | 25.83 pt | 1.08 px | 30.67 pt | **2.5** | 0.33 px | 3 |
| `apple_app_store` | 25.83 pt | 1.06 px | 30.67 pt | **2.5** | 0.56 px | 1 |
| `apple_dev` | 25.83 pt | 1.10 px | 30.67 pt | **2.5** | 0.31 px | 2 |
| `apple_fitness` | 37.83 pt | 1.81 px | 45.00 pt | **2.55** | 0.66 px | 16 |
| `apple_music` | 37.83 pt | 2.16 px | 46.00 pt | **2.6** | 0.69 px | 1 |
| `apple_tv` | 37.83 pt | 2.16 px | 46.00 pt | **2.6** | 0.69 px | 1 |

Девять продуктов, одна батарея замеров, 31 подгонка. Суперэллипс точнее дуги
**во всех девяти случаях без исключения** — в 1.7–3.5 раза по RMSE, — а
показатель держится в узком коридоре **n = 2.5–2.6**. Десятый, Calendar, не дал
ни одной карточки, проходящей отбор: у него в кадрах сетки и формы, а не
сгруппированный список.

**Радиус зависит от роли, показатель — нет.** Числа разбиваются на две семьи:
**24–26 pt** у карточки с врезкой 16 и **37.8 pt** у полноширинной карточки
(врезка 0, ширина 393). Форма угла при этом одна и та же. Значит непрерывность —
свойство **угла**, а радиус — свойство **роли элемента**; это разные решения, и
переносить их надо порознь.

⚠️ **Правка чисел.** Прежняя запись опиралась на один кадр (`apple_id` с.4) и
на довод «круг дал бы 19.3 px на 20-м пикселе, замер даёт 24». Замер 24 был
снят по точному совпадению цвета — такой способ завышает отступ примерно на
2 px, потому что считает антиалиасинг холстом. Субпиксельный замер того же
угла даёт **22.1 px**, а не 24. Довод был верен по направлению и завышен по
величине; радиус **22.3 pt заменён на 24–26 pt** (эквивалент дуги) —
и это, в отличие от прежнего числа, сходится с ЗКН-Д018, где карточка
зафиксирована на 24.

⚠️ n ≈ 2.5 — это **мягкая** непрерывность, а не «яблочный сквиркл» из
презентаций, где показатель называют 4–5. Числа сняты с кадров, а не
с презентаций.

Замер воспроизводится, а не берётся на слово — инструмент лежит в дереве:

```bash
python3 tools/assets/offload.py restore --class ios26-refs   # мастера с зеркала
python3 tools/ios26-corner.py /tmp/apple_files.pdf           # сам найдёт карточку
python3 tools/ios26-battery.py /tmp/apple_files.pdf          # вся батарея разом
```

| Элемент | Радиус | Источник |
|---|---|---|
| **Карточка группы** | **24** | 📐 24.2–25.8 pt по четырём продуктам · ЗКН-Д018 фиксирует 24 |
| Меню, лист, алерт | 20 | 📐 `apple_files` с.11 · `apple_dev` с.7 |
| Карточка-герой с изображением | 16 | 📐 `apple_dev` с.2 |
| Кнопка, чип, поле | **капсула** = `height / 2` | 📐 |
| Плавающий таб-бар | **капсула** | 📐 `apple_files` с.4 — профиль совпал с окружностью `h/2` до пикселя |
| Плитка иконки 29 pt | 7 | 🍎 |

```css
border-radius: 24px;
/* Точное непрерывное скругление там, где браузер умеет: */
@supports (corner-shape: squircle) {
  .card { corner-shape: squircle; border-radius: 16px; }
}
```
⚙️ Круглый `border-radius: 24px` визуально занимает ту же зону, что непрерывный
16 pt, — поэтому **в обычном CSS берётся 24**, а не 16. Подмена радиуса без
подмены формы даёт «пузырь» вместо яблочного угла.

**Правило концентричности** 🍎: вложенный элемент получает
`R_внутр = R_внешн − отступ`. Карточка 24, поле 16 → миниатюра внутри 8.
Одинаковый радиус у вложенных углов читается как ошибка вёрстки.

21.0 + 31 = 52; 28.0 + 24 = 52. Мерить надо было середину, а не кромку.

на деле он в кластере таб-бара (21.0 / 62.0, 11 кадров). И сами числа 21.3 и
28.3 завышены на треть пикселя: кромка бралась по яркости с ореолом стекла,
а не по материалу. Чистые значения — 21.0 и 28.0.

поиск — не пятый таб, а действие.

### 5.14 Медиа-витрина (App Store · TV · Music)

📐 Перепись выделяет витрины в отдельный класс экранов — по числам, не по виду:

| Признак | Витрина | Список |
|---|---|---|
| Врезка | **20** | 16 |
| Подложка под медиа | `#121212` | — |
| Шаг строки | **87** (обложка 16:9) · **64** (миниатюра 48) | 48 · 50 · 54 |
| Холст | `#000000` — **не смягчается** под изображением | `#000000` |

Из этого следуют два правила, которые видно в числах:

1. **Витрина не прячет чёрный.** Ни у App Store, ни у TV, ни у Music холст не
   уходит в тёмно-серый ради «мягкости»: постер работает контрастом к чистому

### 5.20 Кинетика нижнего слоя — состояния и переходы

**Что здесь можно доказать, а что нельзя.** Анимацию со статичного кадра снять
невозможно: ни длительность, ни кривую. Но **состояние** снять можно, и если в
наборе есть кадры одного продукта в разных состояниях, то переход — это вектор
между двумя измеренными точками. Ниже — только это: точки и векторы. Времени и
кривых здесь нет и быть не может (🕳, нужна запись экрана).

Слой разбирался **на сегменты**: не «панель шириной X», а все капсулы в полосе
и зазоры между ними. Одна панель — это уже интерпретация, и она скрывает
механику.

#### Состояние 1 — навигация: табы + отдельный поиск

| Продукт | Капсула | Элементов внутри | Отдельная кнопка | Полоса |
|---|---|---|---|---|
| Apple TV | 📐 `[21 + 281]` | 4 | 📐 `[310 + 62]` | 21 … 372 |
| Music | 📐 `[21 + 281]` | 3 | 📐 `[310 + 62]` | 21 … 372 |
| Apple Developer | 📐 `[21 + 281]` | — | 📐 `[310 + 62]` | 21 … 372 |
| App Store | 📐 `[21 + 351]` | — | **нет** | 21 … 372 |

**Арифметика сходится точно:** 281 + 8 + 62 = **351**, и 21 + 351 + 21 = **393**.
Полоса слоя одна и та же независимо от того, разбита она на две капсулы или
слита в одну. У App Store поиск — пятый таб, поэтому капсула цельная; у TV,
Music и Developer поиск вынесен, и та же полоса делится 281 / 8 / 62.

⚠️ Зазор **8 pt** снят по границе с ореолом; методом материала (раздел 0) тот же
зазор в Notes даёт **12 pt**. Расхождение в 2 pt на сторону — ореол стекла.
Верное число — 12; 8 оставлено как есть, чтобы видна была цена метода.

#### Состояние 2 — вход в раздел: табы сворачиваются, слева встаёт группа

Files даёт обе точки прямо:

| Экран | Капсула | Элементы внутри |
|---|---|---|
| Верхний уровень (Recents · Shared · Browse) | 📐 `[60 + 274]` | 📐 три равных: x 92 (39) · 181 (34) · 266 (34), шаг **87 pt** |
| Внутри папки (`‹ Downloads`) | 📐 `[40 + 294]` | 📐 **широкий 90 pt вплотную к левому краю** + 15 · 40 · 53 |

Капсула выросла на 20 pt **влево** (правый край остался на 334), а внутри
вместо трёх одинаковых табов встал **один широкий элемент у левого края** и
набор узких справа. Это и есть сворачивание: разделы уступают место контексту,
а сами уходят в один элемент слева.

📐 Адреса: развёрнутое — `apple_files` с.2, 3, 4, 11, 15; свёрнутое — с.7, 8, 9.

#### Состояние 3 — плеер: слой отдаётся управлению

Apple TV, семь кадров:

| Что | Навигация | Плеер |
|---|---|---|
| Первая капсула | 📐 281 pt, 4 элемента | 📐 **51 pt, 1 элемент** |
| Дальше | 📐 + 62 (поиск) | 📐 + 70 + 140 |
| Полоса | 21 … 372 (**351**) | 21 … 298 (**277**) |
| Высота | 📐 62 pt | 📐 38 pt |
| Заливка | 📐 `#181818` | 📐 `#111111` |
| Середина над низом | 📐 52 pt | 📐 56 pt |

Табы схлопываются с 281 pt до **51 pt с одним знаком**, освободившееся место
занимают две контекстные капсулы (70 и 140). Слой не прячется — он **меняет
хозяина**: навигация уступает управлению воспроизведением.

📐 Адреса: `apple_tv` с.6, 9, 12, 27, 28, 29, 30 — плеер; с.1, 3, 14, 17, 20,
23, 24, 25, 26 — навигация.

#### Состояние 4 — перемотка: слой становится одной дорожкой

📐 `apple_tv` с.31: капсула `[21 + 351]`, внутри **два элемента по краям**
(x 38 w 25 и x 322 w 33), середина пуста — это шкала. Полоса вернулась к 351,
но содержимое стало одной дорожкой с двумя значениями по концам.

#### Состояние 5 — действие вместо навигации

Fitness показывает, что слой может перестать быть навигацией совсем:

| Кадр | Капсула | Внутри |
|---|---|---|
| 📐 с.11, 19, 21–24 | `[60 + 274]` | три таба, шаг 88 pt |
| 📐 с.12 | `[38 + 317]` | **один элемент 120 pt по центру** |
| 📐 с.13 | `[38 + 317]` | два элемента: 91 + 61 |
| 📐 с.14 | `[16 + 361]` | один элемент 51 pt по центру |
| 📐 с.6 | `[20 + 354]` | **один элемент во всю капсулу** |

Ширина капсулы меняется от 274 до 361 pt, и **во всех состояниях она
центрирована**: врезки 60/59 · 58.7/57.3 · 38/38 · 19.7/19.3 · 16/16.

#### Состояние 6 — второй этаж

📐 Music: мини-плеер `[28 + 48] + [88 + 277]` — квадрат обложки **48 pt** у
левого края и полоса содержимого. Стоит **над** таб-баром через 8 pt (5.17).

#### Что из этого следует

1. **Постоянна ось, а не отступ.** 52 pt держится во всех состояниях
   навигации; у плеера своя — 56 pt.
2. **Полоса слоя — константа продукта, а не экрана.** У TV, Music, Developer и
   App Store это 351 pt при врезке 21. Делится она по-разному, сумма не меняется.
3. **Капсула всегда центрирована** — проверено на пяти ширинах Fitness.
4. **Число табов не задаёт ширину.** У TV в 281 pt помещаются 4 элемента с
   шагом 67, у Music — 3 с шагом 87. Ширину задаёт содержимое, а не счёт.
5. **Слой имеет как минимум шесть состояний**, и они не декоративны: навигация,
   вход в раздел, плеер, перемотка, действие, второй этаж.

🕳 Не снято и снято быть не может со статики: длительность и кривая перехода,
порядок появления элементов, поведение при прокрутке, отклик на нажатие.

### 5.21 Перепись надписей — кегль, вес, цвет

**Вес посчитан, а не увиден.** У шрифта вес — это толщина штриха; при известной
высоте заглавной отношение «стебель / cap» делит начертания надёжнее глаза.
Толщина стебля берётся модой длин горизонтальных отрезков чернил внутри литеры:
строка пересекает вертикальный штрих ровно на его толщине, и мода устойчива к
скруглениям и антиалиасингу.

⚙️ Границы классов: Regular < 0.145 ≤ Medium < 0.168 ≤ Semibold < 0.192 ≤ Bold.

Перепись: **837 строк текста** на пяти продуктах (`apple_id`, `apple_files`,
`apple_notes`, `apple_dev`, `apple_wallet`).

#### Шкала: кегль × вес

| cap-height | Вес | Строк | ⚙️ Роль |
|---|---|---|---|
| 📐 **10.33 pt** | Regular | **194** | основная строка списка — самый частый класс корпуса |
| 📐 10.00 pt | Regular | 63 | строка в плотном списке |
| 📐 8.67 pt | Regular | 42 | вторичная подпись под строкой |
| 📐 9.00 pt | Medium | 41 | подпись элемента управления |
| 📐 9.00 pt | Semibold | 36 | подпись активного элемента |
| 📐 8.00 pt | Medium | 34 | мелкая подпись |
| 📐 16.00 pt | Medium | 34 | заголовок раздела |
| 📐 10.33 pt | Bold | 22 | выделенная строка |
| 📐 17.33 pt | Regular | 19 | крупный текст |
| 📐 13.33 pt | Medium | 18 | подзаголовок |
| 📐 19.33 pt | Regular | 11 | заголовок экрана |
| 📐 22.67 pt | Bold | — | крупный заголовок (`apple_id` с.1) |

**Один и тот же кегль несёт разный вес — это и есть выделение.** cap 10.33
встречается Regular (194 строки), Medium (19) и Bold (22): различие между
обычной строкой, подписью и выделенной — не размер, а **штрих**. cap 9.00
делится ещё резче: Medium 41 против Semibold 36.

⚠️ Перевод cap → кегль остаётся ⚙️ и приблизительным. При ratio 0.705 (SF Pro)
cap 10.33 даёт ≈ 14.7 pt, но крупный заголовок `apple_id` с.1 меряется как
22.67 pt cap ≈ 32.2 pt при каноне Large Title 34 — недобор около 5 %. Причина
в том, что мода «высокого кластера» ловит и выносные, которые выше заглавных.
**Сравнивать классы между собой можно, называть абсолютный кегль — нет** (🕳).

#### Веса: их четыре, и Regular держит большинство

| Вес | Строк | Доля |
|---|---|---|
| 📐 Regular | 493 | 59 % |
| 📐 Medium | 178 | 21 % |
| 📐 Semibold | 86 | 10 % |
| 📐 Bold | 80 | 10 % |

⚙️ Пятого веса в корпусе нет. Light, Thin и Heavy не встречаются ни разу — это
не «недобор выборки», а решение: интерфейс держится на четырёх начертаниях.

#### Цвет надписи: три уровня, и они не нейтральные

📐 Замер по плоским (несцветным) пикселям текста, четыре продукта:

| Уровень | Замер | 🍎 Токен |
|---|---|---|
| Основной | 📐 `#F8F8F8` … `#FFFFFF` | `label` = `#FFFFFF` |
| Вторичный | 📐 `#888890` · `#909088` · `#989890` | `secondaryLabel` = `rgba(235,235,245,0.60)` |
| Третичный | 📐 `#707070` · `#585858` · `#484848` | `tertiaryLabel` = `rgba(235,235,245,0.30)` |

**Серый у Apple с синим подтоном, а не нейтральный.** В замерах вторичного
уровня синий канал систематически выше зелёного и красного (`#888890`,
`#909088`) — это след базы `#EBEBF5`, приглушённой альфой, а не серой краски.
Нейтральный `#8E8E8E` вместо `#8E8E93` даст едва заметный, но систематический
сдвиг тепла по всему интерфейсу.

### 5.22 Знаки: габарит и толщина штриха

Полный обход всех знаков в шапках дал кашу, и это стоит записать как результат:
детектор рвёт знак на части (у лупы отдельно круг и ручка), а квадратные литеры
попадают в выборку наравне со знаками. Числа из такого прохода публиковать
нельзя. Мерить надо там, **где колонка знака известна заранее** — в строке меню
и в строке списка.

| Что | Габарит | Штрих | Адрес |
|---|---|---|---|
| Знак в строке меню | 📐 **14.0 × 14.0 pt** (42 × 42 px) | 📐 **1.33 pt** (4 px) | apple_notes с.7 |
| Мелкий знак в меню | 📐 11.7 × 11.7 pt (35 × 35 px) | 📐 1.00 pt (3 px) | apple_notes с.7 |
| Галочка в подменю | 📐 **6.7 × 10.0 pt** (20 × 30 px) | 📐 1.33 pt (4 px) | apple_notes с.8 — три строки подряд одинаково |
| Плитка иконки в строке списка | 📐 18.0 × 18.0 pt (54 × 54 px) | заливка 0.55 | apple_id с.1 |

**Штрих квантуется по пикселю, а не масштабируется плавно.** Во всех замерах он
равен ровно 3 или 4 физическим пикселям — 1.00 или 1.33 pt при @3x. Промежуточных
значений нет ни разу. Значит толщину знака нельзя задавать долями: при @3x она
ляжет на целый пиксель, и знак, нарисованный в 1.2 pt, отрисуется как 1.33 —
но с грязным краем.

**Галочка — не знак, а другой класс.** 6.7 × 10.0 pt: она у́же и выше знака,
пропорция 2:3 против 1:1. Ставить её в ту же сетку, что иконку, нельзя.

⚙️ Отношение штрих/габарит: 1.33 / 14.0 = **0.095**, у мелкого 1.00 / 11.7 =
0.085. То есть штрих растёт медленнее габарита — крупный знак выглядит тоньше,
и это ровно то поведение, которое даёт SF Symbols при оптическом масштабировании.

🕳 Не снято: заливка против контура (когда знак активного таба становится
залитым), оптические размеры SF Symbols и сетка построения самих символов.
Для первого нужны парные кадры одного таба в двух состояниях.

### 5.23 Кнопка-капсула: высота постоянна, ширина по содержимому

Кнопку я нашёл не с первого раза, и промах стоит записать. Первые два прохода
искали заливку **светлее холста** — и не находили ничего. Кнопка «Get» в App
Store залита `#1C1C1C` на `#000000`: она темнее любого разумного порога
яркости. Порог отсекал её как фон.

Второй промах: ширина мерилась прогоном однородной заливки, а надпись внутри
капсулы этот прогон **разрывает**, и получались куски по 31–39 pt вместо целой
кнопки. Габарит надо брать по границе с холстом, а не по заливке.

| Параметр | Замер | Адрес |
|---|---|---|
| **Высота** | 📐 **32.0 pt** — на всех 18 замерах, разброс нулевой | apple_app_store с.5, 10, 22 |
| Радиус | ⚙️ **16.0 pt = h / 2** — капсула | там же |
| Ширина | 📐 **44.7 … 62.0 pt** — по надписи | там же |
| Заливка | 📐 `#1C1C1C` на чёрном холсте | там же |

**Высота кнопки — константа, ширина — переменная.** Восемнадцать капсул в
четырёх кадрах дали ровно 32.0 pt без единого отклонения, при том что ширина
гуляет на 17 pt. Значит в системе кнопка задаётся высотой и внутренними
отступами, а не размером целиком; радиус при этом не назначается, а следует из
высоты.

⚠️ Правый край капсул не совпал: 358.3 … 367.0 pt. По одному правому отступу
они не выровнены, и объяснения у меня нет — это либо разные контексты строки,
либо ореол, взятый по-разному. Записываю как есть (🕳), а не подгоняю под
красивое число.

### 5.24 Шеврон и сетка: величины ложатся на пиксель, а не на 8 pt

#### Шеврон раскрытия

Два предыдущих прохода промахнулись, потому что я задавал координаты рукой.
Верный способ: полосу строки задают **разделители**, а сам шеврон ищется в
правом краю карточки **по пропорции** — он у́же и выше знака.

| Параметр | Замер | Адрес |
|---|---|---|
| Габарит | 📐 **5.67 × 9.67 pt** (17 × 29 px) | apple_id с.1, 3 |
| Пропорция в/ш | 📐 **1.71** | шесть замеров, ширина совпала во всех шести |
| Правый край | 📐 358.0 … 359.7 pt | ⚙️ = правый край карточки 377 − 16 |

⚙️ Замыкающий элемент отбит от кромки карточки на **16 pt** — столько же,
сколько карточка отбита от края экрана. Врезка одна и та же снаружи и внутри.

#### Сетка: её нет в точках, она есть в пикселях

Двадцать семь величин, снятых за сессию — от толщины штриха 1.33 pt до полосы
слоя 351 pt, — проверены на кратность:

| Проверка | Сходится |
|---|---|
| Целое число **физических пикселей** (× 3 при @3x) | 📐 **27 из 27** |
| Кратно 4 pt | 📐 7 из 27 |
| Кратно 8 pt | 📐 5 из 27 |

**Сетки в точках у iOS 26 нет.** Шаг строки 49.0 pt, врезка разделителя
68.7 pt, плотная строка 34.7 pt, cap строки 10.33 pt, шеврон 5.67 × 9.67 pt —
ни одна из этих величин не кратна ни четырём, ни восьми. Зато **все двадцать
семь без исключения — целые пиксели**: 147, 206, 104, 31, 17, 29.

Отсюда практическое следствие, и оно противоречит привычке, принесённой из
вёрстки: **проектировать на сетке 8 pt значит промахиваться в четырёх случаях
из пяти**. Величина назначается в пикселях устройства и только потом переводится
в точки — поэтому в точках она и выглядит «некруглой». В CSS это значит шаг
**1/3 pt** при @3x, а не 4 и не 8.

⚙️ Обратная проверка: те семь величин, что кратны 4, — это как раз крупные
конструктивные размеры (16, 32, 48, 52, 12), где круглое число в точках и
целое в пикселях совпадают. Кратность там **следствие**, а не правило.

### 5.25 Поиск: что происходит при фокусе

Пара кадров одного экрана в двух состояниях — `apple_notes` с.1 (покой) и с.2
(поиск активен). Мерилась не «форма поля», а **вся раскладка**, потому что
меняется не поле, а экран целиком.

| Что | Покой (с.1) | Фокус (с.2) | Δ |
|---|---|---|---|
| Верх первой карточки | 📐 **229.3 pt** | 📐 **96.0 pt** | 📐 **−133.3 pt** |
| Ширина карточки | 📐 361.0 pt | 📐 361.0 pt | без изменений |
| Врезка | 📐 16.0 pt | 📐 16.0 pt | без изменений |
| Плавающий нижний слой | 📐 есть, верх 777.0 pt | 📐 **нет** | убран |
| Клавиатура | 📐 нет | 📐 верх **517.0 pt** | +335.0 pt снизу |

**Схлопывается шапка, а не поле.** Содержимое поднимается на **133.3 pt** —
ровно на высоту блока с крупным заголовком, который при фокусе убирается.
Горизонтальная геометрия при этом не трогается вовсе: врезка 16 и ширина 361
остаются те же. Движение только по вертикали.

**Нижний слой уходит целиком.** В покое там плавающая панель, при фокусе её нет:
её место занимает клавиатура. Это седьмое состояние слоя к шести из 5.20 —
**отсутствие**.

**Клавиатура: верх 517.0 pt, высота 335.0 pt.** 📐 Совпало на трёх продуктах
(`apple_notes` с.2, `apple_music` с.25 и 26, `apple_wallet` с.8) — это опора для
вёрстки: при поднятой клавиатуре содержимому остаётся ровно до 517.

⚙️ Арифметика видимой области: в покое содержимое живёт от 229.3 до 777.0 —
**547.7 pt**; при фокусе от 96.0 до 517.0 — **421.0 pt**. То есть фокус поиска
**отнимает 126.7 pt**, несмотря на убранную шапку: клавиатура забирает больше,
чем возвращает схлопнувшийся заголовок. Список при фокусе короче, а не длиннее —
это против интуиции и это надо закладывать в вёрстку.

### 5.26 Полки: период берётся автокорреляцией

Прямой замер плиток не работает: у полки содержимое разное в каждой плитке, и
детектор границ выдаёт ширины 74, 101, 133 pt для одного и того же ряда. Но
полка **периодична**, а период меряется автокорреляцией профиля содержимого —
она устойчива к тому, что нарисовано внутри.

| Полка | Период | Корреляция | Адрес |
|---|---|---|---|
| Постеры | 📐 **112.3 pt** | r = 0.80 | apple_tv с.21 |
| Плитки плейлистов | 📐 **153.0 pt** | r = 0.66 | apple_music с.12 |

⚙️ Две полки — не закон. Но метод рабочий: там, где корреляция выше 0.6, период
устойчив к сдвигу полосы замера на десятки точек. Где ниже 0.3 — полки в этой
полосе нет, и число оттуда брать нельзя (проверено: те же кадры в других полосах
дают r ≈ 0.02 и бессмысленные 40 pt).

🕳 Общий закон шага полки не выведен: нужны замеры по всем полкам всех витрин.

### 5.27 Контраст и зоны нажатия — счёт по снятым числам

Здесь ничего не меряется заново: берутся цвета из 2.6 и 5.21, поверхности из
5.19, габариты из 5.22–5.24 — и считаются.

#### Контраст: одна и та же краска на трёх поверхностях читается по-разному

### 5.28 Интерлиньяж — и два замера, которые не получились

#### Шаг строки в абзаце

Считался по переписи 5.21: расстояние между соседними строками **одного класса**
(разный cap — разный блок, такие пары отброшены). 185 пар на трёх продуктах.

| cap-height | Шаг | Отношение | Пар |
|---|---|---|---|
| 📐 **10.33 pt** | 📐 **19.00 pt** | **1.84** | 65 |
| 📐 9.00 pt | 📐 16.00 pt | 1.78 | 32 |
| 📐 12.67 pt | 📐 22.00 pt | 1.74 | 18 |
| 📐 8.67 pt | 📐 16.00 pt | 1.85 | 13 |
| 📐 10.00 pt | 📐 21.00 pt | 2.10 | 13 |

📐 Медиана отношения по всем 185 парам — **1.84**.

**Независимая проверка перевода кегля.** Раздел 5.21 оговаривал, что cap → кегль
остаётся приблизительным. Интерлиньяж даёт способ проверить его со стороны: при
cap/кегль = 0.705 отношение 1.84 превращается в **line-height / кегль = 1.30**
(1.84 × 0.705 = 1.297). Канон Apple для основного текста — 17 pt при высоте
строки 22, то есть **1.29**. Два независимых замера сошлись на третьем знаке —
значит перевод кегля не сбит, а недобор 5 % в 5.21 объясняется выносными в моде
кластера, а не ошибкой коэффициента.

⚙️ Для CSS: `line-height: 1.30` от кегля, или `1.84 × cap`.

#### Два замера, которые не получились

Записываю, потому что молчащий провал вреднее: следующий заход повторит его.

**Тумблер.** Искал по системному зелёному `#30D158` — нашёл три «тумблера»
62.7 × 26–28 pt в Календаре. Это **не тумблеры, а зелёные блоки событий**:
у настоящего переключателя iOS пропорция около 1.65, а здесь 2.24–2.41. Цвет
оказался недостаточным признаком: событие и тумблер красятся одинаково. Нужен
признак формы — капсула с белым кругом внутри у одного из краёв.

**Кольца прогресса.** Маска «зелёный» собрала весь зелёный кадра, и центр
масс лёг между разными объектами: радиус вышел 103…348 pt при «толщине» 245.
Кольцо нельзя брать цветом — нужен разбор на связные области и проверка, что
точки лежат на окружности (разброс радиуса от центра мал), а не размазаны.

🕳 Оба пункта остаются открытыми: B14 (тумблер, слайдер, шаговик,
сегментированный контрол) и B24 (кольца и графики).

### 5.29 Кольца прогресса — взяты признаком окружности

Заход по цвету провалился (5.28): маска собрала весь насыщенный цвет кадра, и
центр масс лёг между разными объектами. Верный признак не цвет, а **форма**:
разбор на связные области, затем проверка, что точки лежат на окружности —
отклонение радиуса от центра менее 0.22 от среднего. Всё остальное отсеивается.

| Кольцо | Радиус | Толщина дуги | Повторов | Адрес |
|---|---|---|---|---|
| Малое | 📐 **6.7 pt** | 📐 **4.1 pt** | 5 на кадре, значения совпали | apple_fitness с.5 |
| Среднее | 📐 **16.4 pt** | 📐 **7.3 pt** | 3 на кадре, совпали | apple_fitness с.9 |
| Крупное | 📐 29.5 pt | 📐 16.8 pt | 1 | apple_fitness с.9 |

⚙️ Внешний диаметр: малое ≈ 17.5 pt, среднее ≈ 40.1 pt, крупное ≈ 75.8 pt.

⚙️ Отношение толщины к радиусу — **0.45 … 0.61**, константой не является.
Толстое кольцо не масштабируется из тонкого: мелкое кольцо относительно толще
(0.61), крупное тоньше (0.57 при вдвое большем радиусе). Это то же поведение,
что у штриха знака в 5.22 — крупный элемент относительно тоньше.

#### Тумблер: три захода, ни один не сработал

1. По цвету `#30D158` — нашлись зелёные блоки событий Календаря, не тумблеры
   (пропорция 2.24–2.41 против канонических ≈ 1.65).
2. По капсуле с белым кругом — детектор оказался слишком тяжёлым, снят по времени.
3. По белому кругу с порогом `min(RGB) > 200` в правой части экрана — **не найден
   ни один**. Значит ручка тумблера не чисто белая: у неё тень и смягчённый край,
   и жёсткий порог её срезает.

🕳 B14 остаётся открытым. Следующий заход должен искать ручку **относительным**
порогом — светлее окружающей капсулы на столько-то, — а не абсолютным.
Записываю это здесь, чтобы четвёртый заход не повторил третий.

### 5.30 Чип-фильтр: та же логика, что у кнопки, но ступенью выше

| Параметр | Замер | Адрес |
|---|---|---|
| Высота | 📐 **35.0 pt** — шесть совпадений, разброс нулевой | apple_app_store с.23, 29 |
| Заливка | 📐 **`#2C2C2C`** | там же |
| Ширина | 📐 59.7 … 145.3 pt — по надписи | там же |
| Радиус | ⚙️ 17.5 pt = h / 2 | капсула |

**Чип и кнопка устроены одинаково и различаются намеренно.** У обоих высота
постоянна, ширина по содержимому, радиус равен половине высоты. Но кнопка —
32.0 pt на заливке `#1C1C1C`, а чип — **35.0 pt на `#2C2C2C`**: выше на 3 pt и
светлее на ступень. Чип лежит **на** поверхности как фильтр списка, кнопка —
**в** строке как действие. Разница в три точки и одну ступень заливки — это
разница ролей, а не небрежность.

⚠️ В выборку попали заливки `#592C29`, `#3C1F5F`, `#B9A4A9` — это **обложки**,
а не чипы: детектор ловит любую капсулу подходящего размера. Отсеяны по признаку
единой заливки: настоящие чипы дали один и тот же `#2C2C2C` шесть раз подряд,
обложки — шесть разных цветов.

## 6. Материал — Liquid Glass

🍎 Материал iOS 26 — **линза**, а не матовая плёнка. Он преломляет и
**увеличивает** то, что под ним, даёт спектральную кайму по кромке и блик,
едущий за прокруткой. Это уже записано в ЗКН-Д016 и реализовано в таб-баре;
здесь — общее правило и рецепт для остальных поверхностей.

### 6.1 Слои (порядок обязателен)

```
1. backdrop   — blur + saturate  (материал)
2. tint       — белый с малой альфой (тон)
3. refraction — кольцо-маска по кромке: увеличение + аберрация
4. highlight  — inset-блик сверху (спекуляр)
5. shadow     — внешняя тень (отрыв от холста)
```

### 6.2 Рецепт CSS (тёмная тема)

```css
.glass {
  background: rgba(255, 255, 255, 0.06);
  -webkit-backdrop-filter: blur(24px) saturate(180%);
          backdrop-filter: blur(24px) saturate(180%);
  box-shadow:
    inset 0 0.5px 0 rgba(255, 255, 255, 0.18),   /* спекуляр сверху */
    inset 0 -0.5px 0 rgba(255, 255, 255, 0.06),  /* нижняя кромка   */
    0 8px 24px rgba(0, 0, 0, 0.40);              /* отрыв           */
  -webkit-user-select: none;                      /* ЗКН-Д016        */
}
@supports not ((backdrop-filter: blur(1px)) or (-webkit-backdrop-filter: blur(1px))) {
  .glass { background: #181818; }                 /* 📐 замер бара   */
}
```

Соответствия толщин 🍎 → ⚙️:

| Материал Apple | blur | tint (тёмная) |
|---|---|---|
| ultraThin | 12 px | `rgba(255,255,255,0.04)` |
| thin | 18 px | `rgba(255,255,255,0.05)` |
| **regular** | **24 px** | `rgba(255,255,255,0.06)` |
| thick | 34 px | `rgba(255,255,255,0.09)` |
| chrome (бары) | 30 px | `rgba(255,255,255,0.07)` |

### 6.3 Где стекло можно и где нельзя

**Можно:** таб-бар · шапка · меню · алерт · кнопка поверх медиа · чип на фото.

**Нельзя:** карточка контента · строка списка · фон секции.
Это ЗКН-Д014, и он не отменяется: стекло — материал **контролов**, а не
заливка **контейнеров**. Залитая стеклом карточка читается как дырка.
Поверхность контента = `--color-bg-2` + волосяная линия.

**Не прозрачно — значит не стекло.** Молочная непрозрачность запрещена
(ЗКН-Д016): сквозь капсулу обязана читаться структура контента.

## 7. Движение

| Действие | Длительность | Кривая |
|---|---|---|
| Нажатие (`scale`, цвет) | 100 ms | `ease-out` |
| Тумблер, сегмент, чип | 200–250 ms | пружина |
| Появление меню / листа | 300 ms | пружина от точки касания |
| Переход экрана | 400–500 ms | пружина |
| Морф материала | 350 ms | пружина |

```css
--ease-ios:     cubic-bezier(0.32, 0.72, 0, 1);   /* пружина без перелёта */
--ease-out-ios: cubic-bezier(0.25, 0.10, 0.25, 1);
--dur-tap: 100ms;  --dur-ui: 250ms;  --dur-nav: 450ms;
```

⚙️ `cubic-bezier(0.32, 0.72, 0, 1)` — принятое приближение системной пружины
iOS (`response ≈ 0.35`, `damping ≈ 0.9`): быстрый старт, мягкая посадка,
без отскока. Отскок (`bounce > 0`) в iOS 26 остался только у морфа материала.

Обязательно:
```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after { animation-duration: .01ms !important;
    transition-duration: .01ms !important; }
}
```
