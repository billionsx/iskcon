# РЕЕСТР РЕФЕРЕНСОВ · Apple iOS 26.5 · тёмная тема

> Папка `docs/design/ios26/refs/` — **источник истины геометрии** для домена **Д**
> свода `docs/LAWS.md`. Закон: **ЗКН-Д026**.
>
> Референс — не вдохновение. Это **эталон замера**: любое число в интерфейсе
> (кегль, отступ, радиус, высота, цвет, толщина линии) обязано иметь адрес
> в этой папке либо в `STANDARD_ios26_css.md`, который из неё выведен.

## Как снят материал

| Параметр | Значение |
|---|---|
| Устройство | iPhone, экран **1179 × 2556 px**, масштаб **@3x** |
| Логический холст | **393 × 852 pt** |
| Перевод | `pt = px / 3` — все замеры в документах даны **в pt** |
| Тема | **тёмная**, холст `#000000` (замер: 29–59 % площади всех кадров) |
| Система | iOS 26.5 |
| Дата поступления | 19.07.2026 |

Кадры извлекаются из PDF как есть (JPEG, без пересжатия). При замере цвета
берётся **мода по области**, а не одиночный пиксель: JPEG сдвигает значение
на ±1–2 по каналу (`#1C1C1E` читается как `#1B1B1D`).

**Чем TV отличается.** Треть кадров — **плеер поверх видео**, а не список.
Там своя геометрия: панель управления стоит в 37.0 pt от низа при высоте
38.0 pt, то есть её середина — 56 pt, а не 52, как у плавающего слоя обычных
экранов. Витрина же строится не карточками, а постерами: карточки на тёмном
холсте инструмент здесь не находит вовсе.

## Где физически лежат кадры

**Мастера вынесены на два зеркала** — так велит **ЗКН-Ф025**: медиа-бинарь
тяжелее 512 КБ не коммитится в git (раздувает клон, жжёт минуты CI, историю
потом не почистить). Шесть PDF — это 21 МБ; в дереве им не место.

| Что | Где |
|---|---|
| Мастер-кадры (PDF, полное разрешение) | archive.org + GitHub Releases, класс **`ios26-refs`** |
| Индекс истины | `docs/assets/manifest.jsonl` + `docs/assets/iskcon-onelove-ios26-refs.files.tsv` (SHA-256 на каждый файл) |
| **Контактные листы** `<продукт>.contact.jpg` | **в git**, здесь же — все кадры продукта одной картинкой |

```bash
# поднять мастера локально (скачает с живого зеркала, сверит SHA-256)
python3 tools/assets/offload.py restore --class ios26-refs     # 6 продуктов
python3 tools/assets/offload.py restore --class ios26-refs-2   # 4 продукта
```

### Происшествие 19.07.2026 — релиз `assets-ios26-refs-2` пуст

Роспись Notes начиналась с `restore --class ios26-refs-2` — и он упал.

| | Состояние | Как проверено |
|---|---|---|
| GitHub Releases | 🔴 **релиз создан, ассетов ноль** | `GET /releases` → `assets: []`; скачивание → HTTP 404 |
| archive.org | ❓ из песочницы домен закрыт (403 отдаёт политика сети, не сервер) | заключения о зеркале не делается |
| Манифест | заявляет sha256 и 19 844 027 байт | `docs/assets/manifest.jsonl` |

**Почему это не поймали — не то, что кажется.** Гейт живости зеркал в проекте
есть (`tools/assets/verify.py`, еженедельный `assets-verify.yml`, плюс шаг в
приёмке) — и он **отработал**. Просто его модель считает класс целым, пока жив
хотя бы один адрес: фейловер-то работает. Одно живое зеркало из двух — это
«просела редундантность», предупреждение. **Предупреждение в логе CI не читает
никто.** Вдобавок шаг приёмки стоял как `verify.py || echo ::warning::` —
приписка обнуляла и жёсткий провал тоже.

Класс родился полусломанным, гейт сказал об этом вполголоса, и полголоса
хватило. Из песочницы же деградировавший класс равен потерянному: archive.org
закрыт политикой сети, `restore` проваливается на него и падает в пустой
GitHub.

**Починено в тот же день:**

| Что | Где |
|---|---|
| Пустое зеркало GitHub — теперь ошибка, а не предупреждение: заливка в Releases синхронная, лага у неё не бывает (у archive.org бывает — ему скидка осталась) | `verify.py --strict-gh` |
| Приёмка больше не глушит гейт припиской `\|\| echo` | `ios26-intake.yml` |
| Режим починки: пересобирает класс по `<класс>.files.tsv`, недостающее поднимает **из коммита, который его удалил** (коммит ищется сам), сверяет SHA-256 каждого файла и заливает на оба зеркала. Мастера в дерево не возвращаются — Ф025 цел | `offload.py reupload --class X` |
| Запуск починки из CI + `fetch-depth: 0` (на мелком клоне истории нет, подъём мастеров падает) | `assets-offload.yml`, вход `reupload_class` |

## Правила работы с папкой

1. **Файлы не переименовываются.** На них ссылается `STANDARD_ios26_css.md`
   и `docs/LAWS.md`. Переименование рвёт адрес замера.
2. **Файлы не удаляются.** Референс — доказательство числа. Нет кадра — число
   становится «на глаз». Вынос на зеркала удалением **не является**: мастер
   достижим по манифесту.
3. **Новый продукт — просто положить файл.** Кладёшь `apple_<продукт>.pdf`
   в `refs/` (веб-интерфейс GitHub, кнопка **Add file → Upload files**) — и всё.
   Дальше работает `ios26-intake.yml`:

   ```
   заливка PDF  →  контактный лист (≤512 КБ)
                →  строка в таблицу «Продукты» + раздел-заглушка «роспись — долг 🕳»
                →  мастер уезжает на два зеркала СВОИМ классом, из git убирается
                →  гейт Д026 перепроверяет, что ничего не разошлось
   ```

   Имя обязано быть `apple_<продукт>.pdf` — строчными, латиницей, без пробелов:
   на это имя ссылается стандарт, и приём отказывается принимать другое.
   Свой класс на каждую партию — потому что `offload.py` идемпотентен по классу:
   второй заезд под старым именем молча оставил бы мастер в git.
4. Сверка перед работой — по `STANDARD_ios26_css.md`; кадры открываются там,
   где стандарт молчит.

Гейт: `tools/ios26-lint.py` — файл без строки в реестре и строка без файла
роняют сборку.
