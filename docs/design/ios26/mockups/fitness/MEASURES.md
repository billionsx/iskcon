# Fitness — покадровый обмер под мокапы

Здесь лежит то, чего нет в `FRAMES.md`: **поэлементный** разбор кадра. Общая
перепись отвечает на вопрос «какого класса экран и какие у него константы»,
а мокап требует другого — «из чего сложен ровно этот экран и по каким адресам».

Метки происхождения: 📐 **ЗАМЕР** — снято с кадра · ⚙️ **ВЫВОД** — посчитано из
замеров, спорить можно · 🕳 **ДОЛГ** — из кадра не извлекается.

Единица — pt эталона (кадр 1179×2556 px = 393×852 pt @3x, pt = px/3). В CSS
мокапа та же величина стоит в `px`, потому что при @1x они совпадают, а `pt`
в браузере равен ¹⁄₇₂ дюйма и растянул бы макет в 1.333 раза.

**Поправка к стандарту.** Ширина слоя — не константа. Раздел 5.20 даёт полосу
351 (281 + зазор + 62) для продуктов с отдельной кнопкой поиска; здесь слой
один и его ширина 274.0 следует содержимому. Постоянны **ось 52 и высота 62**,
и они сошлись точно. Ширину слоя надо читать как «по содержимому», а 351 —
как частный случай раскладки с двумя капсулами.

**Долг.** Параметры материала (радиус размытия, насыщение, плотность белого)
из одного кадра однозначно не выводятся: наблюдаемый цвет — результат размытия
конкретной подложки. В мокапе стоит `backdrop-filter` с числами, дающими
снятые `#181E12` и `#3B3F31` на этой подложке; на другой подложке они дадут
другой цвет — это правильно, но проверено только здесь.

### Знаки

Знаки **обведены с кадра**, а не нарисованы: `tools/ios26-trace.py` отделяет знак
от подложки по 50 % материала (тот же порог, что у краёв блоков), находит контур
на полупиксельной сетке и прореживает его по Дугласу–Пекеру с допуском 0.4–0.55 px.
Габарит и положение — 📐, форма — ⚙️: обводка растра не равна исходному вектору
Apple, на кромке остаётся ступень порядка ⅓ pt.

### Как проверено, что мокап совпал

Мокап открывается в headless Chrome при deviceScaleFactor 3 (`tools/ios26-shot.js`)
и вычитается из кадра попиксельно.

| Зона | Средняя \|Δ\| | Доля Δ>24 |
|---|---|---|
| карточка 1 | 1.34 | 1.30 % |
| карточка 2 | 1.86 | 1.49 % |
| карточка 3 | 2.30 | 2.15 % |
| шапка | 4.76 | 2.96 % |
| статусбар | 5.61 | 3.35 % |
| слой | 6.57 | 5.96 % |
| баннер | 7.84 | 5.68 % |
| **весь кадр** | **3.57** | **2.81 %** |

Геометрия блоков: наибольшее расхождение **1.01 pt** (кнопка-слот), баннер 0.01,
карточки 0.35 / 0.49 / 1.00.

**Подменный шрифт.** SF Pro в песочнице нет, и без поправки подставленное
начертание сдвигает каждую строку на 1–3 pt — сверка текста была бы бессмысленной.
Поэтому в мокапе объявлено начертание `SFPro-metric`: оно стоит в цепочке ПОСЛЕ
настоящих имён Apple (на Mac берётся настоящий SF Pro) и приводит запасное
начертание к метрикам SF Pro через `ascent-override: 95%`, `descent-override: 25%`,
`size-adjust: 96.2%` — asc 950 · desc 250 · cap 700 при em 1000.

**Трекинг и верх строки** ⚙️ выведены замкнутым циклом: рендер → замер чернил →
поправка → рендер. Цикл сошёлся к расхождению ≤ 1.0 pt по всем строкам; остаток
±0.33 pt — это один физический пиксель при @3x.

**Где цикл соврал.** Подгонять по ширине можно только длинные строки. На «LTE»
(три знака) цикл упёрся в разницу метрик подставного шрифта и накрутил трекинг
до 1.13 em: в песочнице ширина сошлась, а на настоящем SF Pro строка расползлась
через батарею. Теперь трекинг ограничен ±0.06 em и на коротких строках не
подгоняется вовсе — кегль держится по cap-height (это метрика шрифта, она
переносится), а расхождение ширины записывается как известный остаток подстановки:
«LTE» в песочнице шире эталона на 3.0 pt.

**Статусбар.** Полоски сигнала строятся по замеру, а не обводятся: четыре столбика
шириной 3.33 через 5.33, общий низ 36.0, высоты 9.33 / 10.33 / 12.0 / 13.67 📐.
Батарея — знак из ДВУХ слоёв (белый сегмент заряда и серая часть с цифрой),
обведённых по двум порогам яркости в одном габарите 27.33 × 13.67 📐: одним
контуром её не снять, а строить на глаз значит выдумывать.
