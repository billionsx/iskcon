# Глаза агента и защита данных

## 1. Глаза — `shot.yml`

Агент не видит прод: `brajs.com` вне белого списка его песочницы, а `/play` —
SPA, у которого в HTML пусто. Поэтому сканер работает так:

```
workflow_dispatch → Cloudflare Browser Rendering → ветка `shots` → агент
```

* **Кадр.** `/screenshot`, viewport **393×852 @ deviceScaleFactor 3** ⇒ файл
  **1179×2556** — тот же формат, что у эталонов `IMG_2336…2382`. Значит вся
  измерительная оснастка (`pt = px/3`, `tools/music-passport.py`, чернильная
  методика) работает по нашему рендеру без единой правки, и расхождение с Apple
  считается числом, а не глазом.
* **Геометрия.** `/scrape` возвращает точные `left/top/width/height` по списку
  селекторов (`.pl-art`, `.amx-row`, `.amx-h1`, …). Это конец выводу размеров
  из пикселей: сверяем числа напрямую с каноном.
* **Состояние.** Съёмочный режим `?shot=` в `/play`: `?shot=player&book=bg&ch=1`
  открывает плеер на конкретной главе, `?shot=mini` — только мини. Класс `.shot`
  глушит анимации и переходы, поэтому кадр всегда конечное состояние, а не
  середина полёта (та самая обложка «267 вместо 345»).
* **Доставка.** Результат уходит в служебную ветку `shots` (`latest/` и `arch/`),
  агент читает через `raw.githubusercontent.com` — единственный доступный ему
  канал. В `main` бинарь не попадает, храповик `tracked_big_binary` цел.
  Логи шага тоже пишутся туда файлом: логи GitHub Actions песочнице закрыты (403).

**Что требуется один раз.** Токен `CLOUDFLARE_API_TOKEN` сейчас отвечает
`HTTP 401 code 10000` на Browser Rendering — у него нет этого права. Нужно
добавить в тот же токен разрешение **Account → Browser Rendering → Edit**
(Cloudflare → My Profile → API Tokens → Edit). Больше ничего: сканер запустится
сам. Тариф Workers Paid даёт 10 запросов/с REST — с запасом.

## 2. Бэкапы — `backup.yml`

Ежедневно в 02:17 UTC (и по кнопке):

| Артефакт | Что внутри | Чем восстанавливается |
|---|---|---|
| `d1-<дата>.sql.gz` | полный дамп D1: схема + все данные (export API, при сбое — потабличный дамп) | `wrangler d1 execute <db> --file=d1-<дата>.sql` |
| `repo-<дата>.bundle.gz` | `git bundle --all`: **вся история и все ветки** одним файлом | `git clone repo-<дата>.bundle iskcon` |
| `docs/backups/latest.json` | sha256 и размеры обоих файлов | сверка подмены |

**Склад — Internet Archive** (`iskcone-backup-<дата>`): третий провайдер,
отдельные ключи, собственная история версий. Это принципиально: копия внутри
Cloudflare (Time Travel, R2) спасает от аварии и ошибки, но не от того, у кого
есть доступ к аккаунту. Манифест с контрольными суммами коммитится в `main` —
подменённый файл на складе перестанет сходиться с историей коммитов.

### Честная модель угроз

Полностью «неубиваемой» схемы на одном CI не существует: рабочие процессы держат
ключи, и тот, кто может пушить код, может ими воспользоваться. Поэтому защита
строится слоями:

1. **Разделить полномочия.** Токен деплоя не должен уметь удалять D1, токен
   бэкапа — только читать D1 и писать на склад. Сейчас `CLOUDFLARE_API_TOKEN`
   один на всё (123 использования) — это главный риск.
2. **Защитить `main`.** Branch protection + обязательный ревью: без этого
   изменение рабочего процесса = кража секретов.
3. **Отдельный аккаунт IA** со своим паролем и 2FA, не совпадающими с
   GitHub/Cloudflare.
4. **Холодная копия.** Единственный настоящий воздушный зазор — периодически
   (раз в месяц) скачивать item с archive.org на машину вне обоих аккаунтов:
   ноутбук, внешний диск. Пять минут работы, зато переживает любой захват.
5. **Проверять восстановление.** Бэкап, который ни разу не разворачивали, —
   не бэкап. Раз в квартал: развернуть дамп в отдельную D1 и сверить число
   строк по ключевым таблицам с боевой.

### Наблюдение

Отчёт каждого прогона (лог дампа, лог выгрузки, манифест) ложится в ветку
`shots/backup/` — агент читает его тем же каналом и видит, что ночь прошла
успешно, без доступа к логам Actions.
