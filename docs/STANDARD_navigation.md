# Стандарт навигации (кнопка «назад» и сервисные кнопки)

Документ фиксирует модель навигации gaurangers.com и правила: **куда и при какой
ситуации ведут «назад» и другие сервисные кнопки**. Цель — поведение уровня Apple:
предсказуемое, без «потерь», совпадающее с системной кнопкой/жестом «назад».

## 1. Анализ проблемы (что было сломано)

1. **Два независимых роутера с двумя `popstate`-слушателями.** Глобальный роутер
   (`App.tsx`: `applyPath`/`pathFromState`) и внутрикнижный роутер
   (`BookDetailPage.tsx`: главы/стихи). Пока открыта книга, на ОДНО нажатие «назад»
   срабатывали оба: `App` сбрасывал оверлеи и пересобирал состояние из URL,
   `BookDetailPage` независимо синхронизировал главу/стих. Гонка за один URL →
   рассинхрон, «система теряется».
2. **Счётчики на `history.length`** (`enteredAt`, `histBase`). `history.length` **не
   уменьшается** при `back()` — он считает все записи сессии, включая «вперёд» и
   до-приложенческие. Поэтому критерий «есть куда вернуться» был ложным: после
   возврата к точке входа кнопка продолжала делать `history.back()` и **вылетала из
   приложения** или вела не туда.
3. **`pushState(null, …)`** в книге затирал `history.state`, так что любой индекс
   глубины терялся.

## 2. Модель (единый стек навигации)

Приложение — SPA с **единственным** стеком навигации поверх истории браузера.

- **Единственный владелец истории** — `App.tsx`. Все `pushState`/`replaceState`
  идут ТОЛЬКО через `src/nav.ts`.
- **Глубина приложения** хранится в `history.state.appIdx` и обновляется на каждом
  `popstate` (`navSetIdxFromState`). `canGoBack()` ⇔ `appIdx > 0` — надёжный признак
  «под нами есть запись приложения» (в отличие от `history.length`).
- **Каждый уровень = одна запись истории.** «Назад» снимает ровно один уровень.

Уровни:

| Уровень | Что | Кнопка «назад» |
|---|---|---|
| L0 | Табы (Главная, Книги, Киртаны, Герои, Дхама, Кабинет) | нет (навигация табами; браузерный back → предыдущий таб) |
| L1 | Оверлей: книга, чтец, бхаджан, киртан-исполнитель, каталог бхаджанов, коллекция ачарьев, личность, статья, избранное, админка | снять оверлей → вернуться к табу/родителю |
| L2 | Внутри книги: глава | → оглавление книги |
| L3 | Внутри книги: стих | → глава |

`pushUrl` (новый уровень) увеличивает `appIdx`; `replaceUrl` (тот же уровень)
`appIdx` не меняет; `popstate` читает `appIdx` целевой записи.

## 3. Алгоритм «назад» (единая функция)

```
back():
  если canGoBack():            # под нами есть запись приложения
      history.back()           # → popstate → роутер восстановит предыдущий уровень
  иначе:                       # корневая запись (прямой вход / QR)
      replaceUrl("/"); applyPath("/")   # к логическому родителю (главная), НЕ выходим с сайта
```

Это поведение применяется ВЕЗДЕ: и к сервисной кнопке «‹» оверлеев, и к системной
кнопке/жесту «назад» (это и есть `popstate`).

### Делегирование внутрикнижной навигации
Глобальный `popstate` в `App` **пропускает** события, чей путь лежит внутри
открытой книги (`/book/<work>` или `/book/<work>/…`): за главу/стих отвечает
`BookDetailPage`, дублирование исключено. Когда путь выходит за пределы книги —
глобальный роутер закрывает книгу.

### Внутри книги (BookDetailPage)
- Открытие главы/стиха — `pushUrl` (новый уровень).
- Прыжок стих↔стих — `replaceUrl` (тот же уровень: «назад» от стиха ведёт к ГЛАВЕ,
  не к соседнему стиху).
- «Назад»: обычный случай — `canGoBack()` → `history.back()`; холодный вход прямо на
  главу/стих (записей приложения под нами нет) — спуск на уровень вручную через
  `replaceUrl` (стих → глава → книга → `onBack` → главная).

## 4. Правила сервисных кнопок (куда ведут)

| Кнопка / ситуация | Действие |
|---|---|
| «‹ Назад» на оверлее (книга/личность/чтец/…) | снять уровень (см. алгоритм) |
| «‹» на стихе | → глава |
| «‹» на главе | → оглавление книги |
| «‹» на книге | → экран, откуда открыли (таб/избранное/личность) |
| Прямой вход/QR на глубокий экран, «назад» на корне | → главная (без выхода с сайта) |
| Системная кнопка/жест «назад» (Android/iOS/браузер) | то же, что «‹» (это `popstate`) |
| «✕»/scrim/Esc на модалке (донат, QR, ошибка, шит, инфо-лист) | закрыть модалку; подложку не трогаем |
| Донат (`/donate`) — закрытие | снять запись `/donate` (`history.back`), вернуться откуда пришли; прямой вход → главная |
| Логотип ISKCON ONE LOVE в шапке | на Главную (таб home) |
| Сердце в шапке | открыть «Избранное» (оверлей) |
| Таб нижней навигации | переключить раздел; повторный тап активного — скролл вверх |

## 5. Инварианты (не нарушать)

- Никакого `pushState`/`replaceState` мимо `src/nav.ts`.
- Только ОДИН глобальный `popstate`-слушатель (в `App`). Внутрикнижный слушатель
  `BookDetailPage` синхронизирует только пути внутри своей книги и не трогает
  `appIdx`.
- Никаких решений на основе `history.length`.
- Кнопка «назад» НИКОГДА не выводит пользователя за пределы сайта по своей воле:
  на корневой записи — переход к логическому родителю.
- Модалки не участвуют в стеке навигации (кроме доната, у которого есть адрес).

## 6. Файлы

- `src/nav.ts` — единый слой истории (`navInit`, `navSetIdxFromState`, `pushUrl`,
  `replaceUrl`, `canGoBack`, `navIdx`).
- `src/App.tsx` — глобальный роутер: `pathFromState`/`applyPath`, единственный
  `popstate` с делегированием книги, `goBack`/`navigate`/`openDonate`/`closeDonate`.
- `src/BookDetailPage.tsx` — внутрикнижный роутер глав/стихов поверх общего стека.

## Витрина Богатств — единый обычай (ЗКН-Н007 · Н024 · Н044)

**Порядок витрин — закон** (утв. основателем 13.07.2026):

    Книги · Личности · Бхаджаны · Киртаны · Дхама · Прасад

Порядок живёт в ОДНОМ месте — `BOG_SUBS` (`App.tsx`); рейка `HallTabs` читает
тот же список. Два списка = два порядка = два разных приложения. Гейт:
`nav-audit.py::check_n007` сверяет ОБА.

**Каждая витрина открывается одним и тем же обычаем, сверху вниз:**

| слой | компонент | обязателен | правило |
|---|---|---|---|
| надзаголовок (золотом) | `HubHeader eyebrow` | да | КОНТЕКСТ раздела. НЕ повторяет вкладку: вкладка «Личности» → надпись «Вечные спутники» |
| заголовок | `HubHeader title` | да | имя раздела |
| подзаголовок | `HubHeader subtitle` | да | ориентир: что человек здесь найдёт |
| поиск | `HubSearch` | да | лупа слева · крестик очистки · `Esc` сбрасывает · `Enter` открывает первый результат |
| счётчик найденного | `HubCount` | при поиске | «12 книг» |
| пусто | `HubEmpty` | при поиске | «Ничего не найдено по запросу «…»» + подсказка |

Всё это — ОДИН файл: `ui/HubHeader.tsx`. Рисовать шапку или строку поиска в
обход него запрещено (`nav-audit.py::check_n024`, `::check_n044`).

**Отступы витрине не принадлежат.** Сверху — `HUB_TOP = 28`, живёт в зале.
Сбоку — 16px, живёт в зале. Витрина не ставит своего ни сверху, ни сбоку и не
обрамляет страницу заново (`maxWidth` + `margin: "0 auto"` внутри витрины
запрещены: оболочка приложения уже 480px).

**Шапка принадлежит ВИТРИНЕ, а не её разделу.** В Прасаде `HubHeader` жил внутри
раздела «Книга» — и на «Рецептах», «Подборе», «Кому дорого», «Подношении»
надзаголовка с заголовком не существовало вовсе. Шапка и поиск стоят ВЫШЕ рейки
разделов (Tier-2), а не внутри одного из них.
