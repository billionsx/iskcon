## 0. Что реально означает «автономная работа» (честно)

Чтобы не было ложных ожиданий:

- Я **не работаю в фоне 24/7 и не запускаюсь по расписанию сам.** Я действую, когда вы пишете мне в сессии.
- Но **с подключёнными коннекторами** (Supabase, Cloudflare, GitHub, Vercel) в рамках сессии я могу **сам**: создавать и менять схему БД, выполнять SQL, заводить хранилища/бакеты, управлять ресурсами Cloudflare, читать/писать в репозиторий (если GitHub подключён) и запускать деплой.
- Состояние между сессиями хранится **в репозитории и базе** (а не «в моей памяти»), поэтому каждый шаг фиксируем в git/БД.
- Итог: «автономно» = **в наших сессиях я делаю ручную инфраструктурную и кодовую работу за вас**, опираясь на git/Supabase/Cloudflare как на источник истины.

> **Безопасность — прочитайте до начала (раздел 6):** не вставляйте секретные ключи и токены прямо в чат. Используйте коннекторы и хранилища секретов платформ.

## 2. GitHub — создать репозиторий и дать мне доступ

### 2.1 Создать организацию и репозиторий
1. Зайдите на **github.com** → справа сверху «+» → **New organization** (например `iskcone`). Можно и в личном аккаунте, но организация удобнее для команды.
2. «+» → **New repository** → имя `iskcone` → **Private** (приватный на старте) → создать.

### 2.2 Подключить GitHub-коннектор к Claude (рекомендуемый способ)
1. В приложении Claude откройте **Settings → Connectors**.
2. Найдите/подключите **GitHub**, авторизуйтесь, дайте доступ к организации/репозиторию `iskcone`.
3. Убедитесь, что коннектор **включён для этого чата** (тумблер в окне беседы).

После этого в сессии я смогу читать и коммитить в репозиторий — в том числе **залить туда `PRODUCT_ARCHITECTURE.md`**, чего вы и хотели («документ на репозитарии»).

### 2.3 Альтернатива — токен (если не используете коннектор)
Если предпочитаете токен: GitHub → **Settings → Developer settings → Fine-grained personal access tokens** → создать токен с доступом **только к репозиторию `iskcone`** и правами **Contents: Read & Write**, **Pull requests: Read & Write**. Срок — короткий.
**Не присылайте токен в чат.** Если он понадобится, я подскажу безопасный способ передать его (через секрет деплоя/окружения), а не текстом.

### 2.4 Стартовая структура веток
- `main` — продакшен. `dev` — разработка. Фича-ветки `feature/...`, PR в `dev`.

## 3. Supabase — база данных, авторизация, хранилище

Коннектор уже подключён. Нужно создать проект и убедиться в доступе.

1. Зайдите на **supabase.com → Dashboard**.
2. **New organization** (если ещё нет), затем **New project**:
   - Имя: `iskcone-prod`.
   - **Регион** — ближайший к основной аудитории (для глобального проекта подойдёт EU/Frankfurt; можно начать с региона рядом с вами).
   - Задайте пароль БД (сохраните в менеджере паролей — **не в чат**).
3. После создания во вкладке **Project Settings**:
   - **API** — отметьте, что есть `Project URL`, `anon key` (публичный, безопасен для фронтенда) и `service_role key` (**секретный** — только для сервера, никогда на клиент и не в чат).
   - **Database** — данные подключения для миграций.
4. **Authentication → Providers** — позже включим email, Google, Apple, телефон-OTP (для единого аккаунта преданного).
5. **Storage** — позже заведём бакеты (например `center-photos`, `deity-photos`, `media`).
6. **Включите коннектор Supabase для этого чата.** В сессии я смогу: создавать таблицы и миграции, выполнять SQL, настраивать RLS, заводить бакеты, генерировать TypeScript-типы.

> **RLS (Row Level Security):** для мульти-тенанта включаем RLS на таблицах `center_*` — писать может только админ своего центра, глобальный контент пишут только глобальные редакторы. Это настрою в Фазе 0.

## 4. Cloudflare — домен, CDN, хостинг, хранилище

Коннектор уже подключён. Нужно завести домен и ресурсы.

### 4.1 Домен iskcone.com
1. Зайдите на **dash.cloudflare.com**.
2. **Add a site** → `iskcone.com` → выберите план (Free на старте достаточно).
3. Cloudflare покажет **свои nameservers** — пропишите их у регистратора домена. После проверки DNS управляется в Cloudflare.

### 4.2 Хостинг приложения (Cloudflare Pages)
1. В Cloudflare → **Workers & Pages → Create → Pages → Connect to Git**.
2. Подключите репозиторий `iskcone`, выберите фреймворк **Next.js**.
3. Переменные окружения (Project → Settings → Environment variables): `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY` и серверные секреты (`SUPABASE_SERVICE_ROLE_KEY`, ключи Stripe) — **вводите в интерфейсе Cloudflare, не в чат**.
   - _(Альтернатива: тот же Next.js можно хостить на Vercel — у вас подключён и Vercel-коннектор. Решим в открытом вопросе по хостингу.)_

### 4.3 Хранилище и медиа
- **R2** (Workers & Pages → R2) — бакет, например `iskcone-media`, для картинок/аудио/видео.
- При необходимости **Cloudflare Stream** для видео-трансляций.

### 4.4 Токен/аккаунт для автономной работы
1. Профиль (справа сверху) → **My Profile → API Tokens** или **Manage Account → API Tokens**.
2. Создайте токен по принципу **наименьших привилегий** (например права на Pages, DNS, R2 нужного аккаунта). Запишите **Account ID** (на странице обзора аккаунта).
3. **Включите коннектор Cloudflare для этого чата** — тогда в сессии я смогу управлять Pages, DNS, R2, D1 и т.д.
4. **Не присылайте токен в чат.** Если потребуется — настроим через коннектор или хранилище секретов.

## 5. Как всё связывается вместе

```
GitHub (код)
   └─ push в main / dev
        └─ CI/CD (GitHub Actions)
             └─ деплой → Cloudflare Pages (или Vercel)   ← фронтенд Next.js + PWA
                         │
                         ├─ Supabase  ← Postgres, Auth, Storage, Edge Functions (бэкенд)
                         └─ Cloudflare ← CDN, DNS (iskcone.com), R2 (медиа)
```

- **Секреты** живут в Cloudflare/Vercel (env vars) и в GitHub Actions (Secrets) — **не в коде и не в чате**.
- Источник истины по структуре и схеме — **репозиторий** (`supabase/migrations`, `docs/`).

## 6. Безопасность (обязательно)

- **Никогда не вставляйте в чат:** `service_role` ключ Supabase, API-токены Cloudflare/GitHub/Stripe, пароль БД, приватные ключи.
- Публичные значения (`anon key`, `Project URL`, `Account ID`) — допустимы, но лучше всё держать в хранилищах секретов.
- Предпочитайте **коннекторы** прямой передаче токенов — так доступ управляется и отзывается централизованно.
- Используйте **наименьшие привилегии** и короткие сроки жизни токенов.
- Включайте 2FA на GitHub, Supabase, Cloudflare.
- Если секрет случайно где-то засветился — **отзовите и пересоздайте** его.
